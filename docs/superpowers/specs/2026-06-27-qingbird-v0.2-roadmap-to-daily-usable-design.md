## 二、设计原则

1. **渐进迭代**：版本号延续 v0.2.x，不跳大版本，每个版本可独立发布
2. **i18n 跟随**：每个版本新增用户面向字符串必须配齐 `zh-CN.yml` + `en-US.yml`
3. **q-code 继承**：概念和设计直接移植，Rust 化适配（trait 替代 duck typing，rusqlite 替代 better-sqlite3）
4. **4 门禁不变**：每个 PR 前必须通过 `cargo build` / `cargo clippy --all-targets -- -D warnings` / `cargo fmt --check` / `cargo test`

## 三、版本路线

### v0.2.14 — 基础设施夯实

**目标**：修复当前问题 + 补齐最小工具集 + 流式体验

#### 3.1 修复失败的 mock 集成测试
- `crates/qbird-code-agents/tests/react_loop_with_mock_test.rs` 中的 `test_agent_full_react_loop` 因端口竞争 flaky 失败
- 原因：mock HTTP server 在 `127.0.0.1:0` 绑定，但线程启动顺序导致请求可能发到关闭的 listener
- 修复方案：给 mock server 加就绪信号（oneshot channel 通知端口已绑定）

#### 3.2 补齐缺失工具
- `glob`（glob 文件搜索）：注册到 ToolRegistry，对标 Claude Code 的找文件能力
- `list_dir`（目录列表）：已出现在 `READ_ONLY_TOOLS` 常量中但从未注册
- `web_fetch`（HTTP 抓取）：`Capability` 枚举已定义，实现简单 GET 请求工具
- 所有新工具配 i18n 描述和错误消息

#### 3.3 流式输出（Interactive 模式）
- Provider trait 新增 `stream()` 方法（默认回退：整体返回后一次性输出）
- DeepSeek/OpenAI 流式 API 接入（SSE 解析）
- Interactive 模式支持打字机输出效果
- `stream: false` 保持向后兼容（`--execute` 模式用非流式）

#### 3.4 安全模块接线
- SecurityConfig.allowed_paths 在工具执行时检查路径合法性
- 只读工具（L0）跳过检查，写入/命令工具（L1/L2）执行前校验前缀

#### 3.5 修复 flaky 测试后，全量 113+ 测试绿色

### v0.2.15 — 记忆系统（Memory + Context）

**目标**：移植 q-code 的记忆和上下文管理，替换粗暴截断

#### 4.1 在 `qbird-code-infra` 内新增 memory 模块
- 依赖 `rusqlite`（替代 better-sqlite3，用 bundled feature 避免系统依赖）
- 包含：MemoryManager + ContextManager + tokenizer + overflow + budgeted_read
- 不增加 crate 数，保持 5-crate 结构不变
- `qbird-code-infra/src/lib.rs` 新增 `pub mod memory;`

#### 4.2 MemoryManager（SQLite + FTS5）
- 增量同步：`save()` 检查指纹（MD5/MD5 内容哈希），unchanged 跳过
- FTS5 全文搜索：`search(query, scope)` 关键词检索
- 预算化读取：`readBudgeted(filePath, budgetTokens)` 按 token 预算截断 Markdown 章节
- 溢出检测：4 级压力（0=安全 → 3=危险），预留 buffer 保护输出
- 检查点：`generateCheckpoint(session)` → 保存为 Markdown 文件 + 索引到 FTS5

#### 4.3 ContextManager（Token 预算窗口）
- `addMessage()` → `getMessagesWithinBudget(budgetTokens)` 从后往前保留
- `checkpointIfNeeded()` 触发条件：使用率超过阈值（configurable, 默认 80%）
- 替代现有 `messages.len() > MAX_HISTORY_MSGS` 粗暴截断

#### 4.4 接入 ReactLoop
- ReactLoopConfig 新增 `context_token_limit` / `checkpoint_threshold` 字段
- 每轮迭代后检查 ContextManager 压力等级，触发自动检查和截断
- 所有新增用户字符串 i18n

### v0.2.16 — 规范工作流（SDD Workflow）

**目标**：移植 q-code 的 SDD 四阶段流水线 + Skill 插件体系

#### 5.1 Skill 系统（移入 `qbird-code-agents` 或新建 `qbird-code-skills`）
- `SkillDescriptor`：id, name, description, version, blocking, autoTrigger, inputSchema, outputSchema, tags
- `SkillRegistry`：`HashMap<String, Skill>`，支持 register/list/execute/match
- `SkillResult`：success, output, errors, metrics（durationMs, tokensUsed）

#### 5.2 SDD 四阶段工作流
1. **SDDProposalGenerator**：根据用户需求生成结构化提案，含 HARD-GATE（用户确认后才放行）
2. **SDDReviewSpec**：规范符合性审查（API 完整性、测试覆盖、lint）
3. **SDDReviewQuality**：代码质量审查（异味检测、架构评估、性能）
4. **SDDArchive**：归档已完成提案到 MemoryManager

#### 5.3 与 ReactLoop 集成
- `--execute` 模式：任务进入先自动匹配 SDD 工作流，走完流水线后执行 ReAct 循环
- `--interactive` 模式：斜杠命令 `/sdd` 手动触发，输出流程状态
- HARD-GATE 通过异步 channel + 用户确认提示实现（不阻塞事件循环）

#### 5.4 i18n
- 所有 SDD 阶段描述、审查报告、错误消息配中英双语

---

### v0.2.17 — 日常可用打磨

**目标**：补齐日常使用体验，让 `qingbird --interactive` 成为日常编码助手

#### 6.1 Token 用量展示
- 每次 LLM 调用后显示 prompt_tokens / completion_tokens
- 会话累计用量追踪
- 成本估算（基于当前 provider 的每 token 单价，可配置）

#### 6.2 对话历史持久化
- 会话自动保存到 SQLite（Message 序列化为 JSON）
- `--interactive` 启动时列出最近会话列表，支持选择续聊
- 斜杠命令 `/sessions` 列出/切换/删除会话

#### 6.3 工具增强
- 代码编辑工具（结构化代码替换，类似 claude-code 的 edit 工具）
- 文件 diff 预览（写入前展示变更）
- 工具输出大小限制（防止工具返回巨量内容撑爆上下文）

#### 6.4 Subagent 并发池
- `SubagentPool`：固定数量 worker 线程（mpsc channel 分发任务）
- 多 Subagent 并行执行（大任务拆解）
- 基于 q-code 的 TaskPlan.stages.parallel 设计

#### 6.5 i18n 全覆盖审查
- 审计所有 `tracing::` 日志（开发面向）确保维持英文
- 审计所有 `println!` / `eprintln!` 确保全部走 `t!()`

## 五、已知风险

1. **SQLite 跨平台**：Windows 上需要 `bundled` feature 编译 SQLite C 库，会增加首次编译时间
2. **FTS5 中文分词**：SQLite FTS5 默认 unicode61 tokenizer 对中文支持有限。q-code 和 DIKW-Memory-System（https://github.com/Zhao961215/DIKW-Memory-System）均使用 `tokenize='trigram'` 解决中文问题，Rust 端复用相同策略。trigram 对 CJK 字符天然有效，不需要额外分词器。后续如需更精准的中文搜索可升级为 `jieba-rs`
3. **流式输出复杂度**：DeepSeek/OpenAI 的 SSE 格式不同，需要 provider 各自实现流解析；Anthropic 流格式也不同
4. **测试覆盖**：Memory + Skill + Workflow 需要大量单元测试和集成测试，预计每个版本新增 30-60 测试

## 六、不包含的范围

以下内容在本路线图中不涉及（留给未来 v0.3+）：
- GUI 版本（electron/tauri/web）
- 可视化 DAG 工作流编辑器（WorkflowDslV1）
- 语义向量搜索（q-code 的 P3 FTS5+TF-IDF 混合检索，但 FTS5 基础搜索会实现）
- 多用户/团队协作模式
