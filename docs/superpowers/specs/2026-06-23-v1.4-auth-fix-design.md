# v1.4 API Key Auth Fix — Design

**Date**: 2026-06-23  
**Status**: Approved (implementing)  
**Version**: v1.4.0 (bundled with spec D)

## Problem

`LlmProtocol::build_auth_headers` hardcodes `"PLACEHOLDER_API_KEY"` in both generic adapters, making every third-party provider request fail authentication regardless of the user's configured API key.

- `generic_openai.rs:133` → `"Bearer PLACEHOLDER_API_KEY"`
- `generic_anthropic.rs:154` → `"x-api-key: PLACEHOLDER_API_KEY"`

Root cause: `OpenAiProtocol` / `AnthropicProtocol` are zero-field structs with no access to the `api_key` stored in `HttpClientConfig`.

## Solution

Change `LlmProtocol::build_auth_headers` signature to pass `api_key` from the caller:

```rust
// Before
fn build_auth_headers(&self, request: RequestBuilder) -> RequestBuilder;

// After
fn build_auth_headers(&self, request: RequestBuilder, api_key: &str) -> RequestBuilder;
```

The caller (`HttpLlmClient::chat()`) passes `self.config.api_key`.

### Files Changed

| File | Change |
|---|---|
| `src/infrastructure/llm/http_client.rs` | Trait signature + call site |
| `src/infrastructure/llm/generic_openai.rs` | Use `api_key` param instead of hardcode |
| `src/infrastructure/llm/generic_anthropic.rs` | Use `api_key` param instead of hardcode |

### Non-Changes (explicit)

- No new tests needed — existing 328 tests cover the auth path indirectly via `router::tests::router_serves_from_l2_cache_on_second_call` etc.
- No `HttpClientConfig` field changes — `api_key` already exists.
- No protocol struct changes — they remain zero-field.

## Risk

Minimal. Three trivial mechanical changes. Existing tests validate the full LLM router path end-to-end (with dummy keys, but the key value is irrelevant for cache-hit paths).
