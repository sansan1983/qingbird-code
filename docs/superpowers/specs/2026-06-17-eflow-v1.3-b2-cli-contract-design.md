## 0. 一句话定义

在 spec B1 的 `SlashCommand` trait 实施基础上，**暴露同一份 trait impl 给跨进程 GUI**：6 个原有事件 + 1 个 `SystemReady` schema 冻结 + stdin 协议 5 个 action + exit code 4 档（0/1/2/130）+ stdout 永远 JSON 契约 + stderr 永远人类可读。`eflow session start` 持续运行，GUI 通过 stdin 发指令、stdout 收事件。**TUI 仍是同进程 trait dispatch，零改造**。

## 1. 背景 & 动机

### 1.1 spec A + B1 留下的延伸

spec A 跑通 LLM 抽象，spec B1 跑通 SlashCommand trait 抽象。**两个 spec 都做对了一件事：trait 注册机制**。但 trait 抽象的"主消费者"目前只有 TUI ：

- TUI 同进程 `Arc<Mutex<Concierge>>` 直传 → `handle_input()` → `CommandRegistry::dispatch()` → `SlashCommand::execute()`
- **GUI 不能这么干**——GUI 是独立进程，没法 `Arc<Mutex<>>` 共享内存

### 1.2 用户硬性要求（v1.3 头脑风暴第 2 阶段）

用户在 v1.3 spec A 头脑风暴阶段明确：

> 我建议：spec B 实施时**同时定义一份"CLI 稳定契约"**（哪怕 v1.3 只有一个 GUI 用户，就是你自己）

> 后期 GUI 接入时**直接 spawn eflow 进程**就行，TUI 不用重写

### 1.3 设计哲学（核心原则）

**轻量化内核 + 高标准要求**（用户 2026-06-17 明确）：

- 核心 crate 不写死任何契约细节
- 契约冻结 v1.3.0 起（spec A 起算），GUI 和 eflow 必须同版本部署
- 冻结范围：6 个原有事件 + 1 个 SystemReady + stdin 5 个 action + exit code 4 档 + stdout/stderr 协议
- 加新事件 / action 走 v1.4+ 候选 + CHANGELOG 标注
- **TUI 不跨进程 spawn 自己**（早期错误想法已纠正）

### 1.4 v4.0 §14.4 GUI 接入路径修订

v4.0 §14.4 原文：
> GUI 扩展（v2.0 预留）— 框架：egui 或 iced — 实现 InteractionLayer trait 即可接入，核心层无感知。

**修订（v1.3 起）**：
> GUI 扩展（v2.0）— 跨进程套壳。eflow 暴露 CLI 稳定契约（`eflow session start` + stdin/stdout 协议），GUI 用 stdio + events 通道接入，技术栈不限。InteractionLayer trait 仍保留给 TUI（同进程 trait dispatch），TUI 内部走 spec B1 的 `CommandRegistry::dispatch()` 不走 CLI 契约。

**关键设计决策**：

1. **`cli/` 是独立目录**——`start` / `init` / 5 个 handlers + 2 个 helper（error/output）—— **和 `interaction/slash/builtin/` 共享 trait impl**——TUI 和 GUI 共用 1 份业务逻辑
2. **subcommand 是 trait impl 的"薄包装"**——subcommand 解析 stdin JSON → 构造 `SlashArgs` → 调 `CommandRegistry::dispatch()` → 把输出格式化到 stdout JSON；**业务逻辑零分叉**
3. **`cli/` 依赖 `interaction/slash/`**，但 `interaction/slash/` 不依赖 `cli/`——**保持单向依赖**（符合 v4.0 §3.2 "类型安全的同步接口"原则）
4. **TUI 零改造**——TUI 走同进程 `CommandRegistry::dispatch()`，**不**走 `cli/` 任何代码——v1.3.1 行为零变化

### 3.3 clap subcommand 树

```rust
// src/main.rs 改
use clap::{Parser, Subcommand};

#[derive(Parser)]
#[command(name = "eflow", version, about = "eflow — efficient multi-agent framework")]
struct Cli {
    #[command(subcommand)]
    subcommand: Option<SubCommand>,
}

#[derive(Subcommand)]
enum SubCommand {
    /// 初始化配置向导
    Init,
    /// 启动会话（持续运行，stdout 输出 events NDJSON；stdin 接收指令）
    SessionStart {
        #[arg(long)]
        config: Option<PathBuf>,
        #[arg(long)]
        lang: Option<String>,
    },
}
```

**关键设计决策**：

- **6 个 subcommand 合并为 1 个 `SessionStart`**——`eflow session start` 启动后**持续运行**，5 个 action（send/end/level/lang/help）通过 **stdin JSON 协议**实现
- **不**有 `eflow session send` / `eflow session events` 等独立 subcommand——v1.3 严格单进程单 Concierge，多 subcommand 进程**无法共享内存**（daemon + IPC 是 v1.4+ 工作）
- **Init 是顶层 subcommand**（不挂在 `session` 下）——它**不需要会话**也不持续运行
- **SessionStart 启动后**就是 headless 模式——**不**渲染 TUI

**重大决策修订**：早期设计"6 个独立 subcommand"被推翻，理由：单进程架构 + GUI 套壳的实际形态 = "spawn eflow start，GUI 通过 stdin 发指令，stdout 收 events"。这才是 spec B2 必须显式写明的关键设计。

### 3.5 headless CLI 契约层（`src/cli/`）

#### `output.rs` —— stdout/stderr 协议

```rust
// src/cli/output.rs
pub struct CliOutput;

impl CliOutput {
    /// 输出 JSON 对象到 stdout（**永远只走 stdout**）
    pub fn json<T: Serialize>(value: &T) -> Result<()> {
        let json = serde_json::to_string(value)?;
        println!("{}", json);  // println! 走 stdout
        Ok(())
    }
    
    /// 输出一行 NDJSON（events 流用）——立即 flush
    pub fn ndjson_event<T: Serialize>(event: &T) -> Result<()> {
        let json = serde_json::to_string(event)?;
        println!("{}", json);
        // 立即 flush，让 GUI 实时收到
        use std::io::Write;
        std::io::stdout().flush().ok();
        Ok(())
    }
    
    /// 输出人类可读消息到 stderr
    pub fn info(msg: &str) {
        eprintln!("{}", msg);
    }
    
    /// 输出错误消息到 stderr
    pub fn error(msg: &str) {
        eprintln!("error: {}", msg);
    }
}
```

**关键设计决策**：
- `CliOutput::json` 用 `println!`（stdout）——**不**flush（GUI 读行前不需强制 flush）
- `CliOutput::ndjson_event` 调 `flush()` ——**events 实时性关键**
- `CliOutput::info` / `error` 用 `eprintln!`（stderr）
- **不**直接调 `tracing`——tracing 自己配 `tracing_subscriber` 走 stderr

#### `error.rs` —— exit code 转换

```rust
// src/cli/error.rs
use std::process::ExitCode;

pub fn exit_code(err: &EflowError) -> ExitCode {
    match err {
        // 用户错误：参数非法、文件不存在、KEY 无效等
        EflowError::Config(_) | EflowError::LlmAuthFailed(_) => ExitCode::from(1),
        
        // 系统错误：网络失败、文件 IO 错误、内部错误
        EflowError::LlmProvider(_) | EflowError::RateLimited(_) 
        | EflowError::Internal(_) | EflowError::Io(_) => ExitCode::from(2),
        
        // 其它默认 1
        _ => ExitCode::from(1),
    }
}

/// Ctrl+C 退出码 130（Unix 习惯）
pub fn handle_sigint() -> ExitCode {
    ExitCode::from(130)
}
```

**关键设计决策**：
- **GUI 友好**：GUI 拿到非 0 exit code 时能区分"重试就有用"（系统错误 2）vs"重试也白搭"（用户错误 1）
- 130 = 128 + SIGINT(2)——**Unix 习惯**

#### `start.rs` —— `eflow session start`（持续运行）

```rust
// src/cli/start.rs
pub async fn run(config: Option<PathBuf>, lang: Option<String>) -> ExitCode {
    // 1. 加载 config
    let config_path = config.unwrap_or_else(|| {
        dirs::config_dir().unwrap().join("eflow").join("eflow.yaml")
    });
    let eflow_config = match load_config(&config_path) {
        Ok(c) => c,
        Err(e) => {
            CliOutput::error(&format!("config load failed: {e}"));
            return exit_code(&e);
        }
    };
    
    // 2. 切语言（如果指定）
    if let Some(ref locale) = lang {
        crate::infrastructure::locale::init(Some(locale));
    }
    
    // 3. 构造 LlmRouter（spec A）
    let mut router = match LlmRouter::from_config(&eflow_config, &provider_dir) {
        Ok(r) => r,
        Err(e) => {
            CliOutput::error(&format!("router init failed: {e}"));
            return exit_code(&e);
        }
    };
    
    // 4. 构造 CommandRegistry（spec B1）
    let mut command_registry = CommandRegistry::new();
    command_registry.register(Arc::new(ModelCmd));
    command_registry.register(Arc::new(ProfileCmd));
    command_registry.register(Arc::new(LangCmd));
    command_registry.register(Arc::new(LevelCmd));
    command_registry.register(Arc::new(HelpCmd::new(command_registry_snapshot)));
    command_registry.register(Arc::new(QuitCmd));
    if let Err(e) = command_registry.required_register(&["model", "profile", "lang", "level", "help", "quit"]) {
        CliOutput::error(&format!("command registry init failed: {e}"));
        return exit_code(&e);
    }
    
    // 5. 构造 Concierge
    let mut concierge = Concierge::new(router, command_registry);
    
    // 6. 订阅事件通道
    let event_rx = concierge.subscribe_events();
    
    // 7. 输出 SystemReady 事件（**第一行**）
    let task_id = concierge.task_id();
    if let Err(e) = CliOutput::ndjson_event(&serde_json::json!({
        "event_type": "SystemReady",
        "task_id": task_id,
        "started_at": chrono::Utc::now().to_rfc3339(),
    })) {
        return exit_code(&EflowError::Internal(e.to_string()));
    }
    
    // 8. spawn 2 个任务 + tokio::select 等待
    let exit_code = run_two_tasks(&mut concierge, event_rx).await;
    
    // 9. 优雅退出
    let _ = concierge.shutdown().await;
    exit_code
}

async fn run_two_tasks(
    concierge: &mut Concierge,
    mut event_rx: broadcast::Receiver<Event>,
) -> ExitCode {
    tokio::select! {
        exit_code = task_a_listen_events(&mut event_rx) => exit_code,
        exit_code = task_b_read_stdin(concierge) => exit_code,
        _ = tokio::signal::ctrl_c() => handle_sigint(),
    }
}

async fn task_a_listen_events(
    event_rx: &mut broadcast::Receiver<Event>,
) -> ExitCode {
    loop {
        match event_rx.recv().await {
            Ok(Event::TaskStarted { task_id, description }) => {
                let _ = CliOutput::ndjson_event(&serde_json::json!({
                    "event_type": "TaskStarted",
                    "task_id": task_id,
                    "description": description,
                }));
            }
            Ok(Event::TaskCompleted { task_id, summary }) => {
                let _ = CliOutput::ndjson_event(&serde_json::json!({
                    "event_type": "TaskCompleted",
                    "task_id": task_id,
                    "summary": summary,
                }));
            }
            Ok(Event::TaskFailed { task_id, error }) => {
                let _ = CliOutput::ndjson_event(&serde_json::json!({
                    "event_type": "TaskFailed",
                    "task_id": task_id,
                    "error": error,
                }));
            }
            Ok(Event::RiskEscalated { task_id, from, to }) => {
                let _ = CliOutput::ndjson_event(&serde_json::json!({
                    "event_type": "RiskEscalated",
                    "task_id": task_id,
                    "from": format!("{:?}", from),
                    "to": format!("{:?}", to),
                }));
            }
            Ok(Event::UserInputRequired { prompt }) => {
                let _ = CliOutput::ndjson_event(&serde_json::json!({
                    "event_type": "UserInputRequired",
                    "prompt": prompt,
                }));
            }
            Ok(Event::SystemShutdown) => {
                return ExitCode::from(0);
            }
            Err(broadcast::error::RecvError::Lagged(_)) => continue,
            Err(broadcast::error::RecvError::Closed) => return ExitCode::from(0),
        }
    }
}
```

**关键设计决策**：
- **`start` 不输出独立"task_id" JSON**——task_id 在 `SystemReady` 事件里就有
- **start 启动 Concierge 后没有"开机任务"**——等 stdin 收到 `send` 才有 TaskStarted
- **第一行 stdout 是 `SystemReady`**——GUI 用这个判断"eflow 启动完成"
- **`SystemReady` 是 v1.3 新增的 1 个事件**——v4.0 原有 6 个 + 这个 = 7 个契约事件

#### `init.rs` —— `eflow init`

```rust
// src/cli/init.rs
pub async fn run() -> ExitCode {
    // 1. 构造 LlmRouter（spec A 已有，from_config 退化路径）
    let router = match LlmRouter::from_config(&default_config(), &provider_dir) {
        Ok(r) => r,
        Err(e) => {
            CliOutput::error(&format!("router init failed: {e}"));
            return exit_code(&e);
        }
    };
    
    // 2. 调 Wizard（spec B1 已有）
    match Wizard::run(router).await {
        Ok(()) => ExitCode::from(0),        // 成功
        Err(WizardError::Cancelled) => ExitCode::from(1),  // 用户 Esc 取消
        Err(WizardError::FinalizeFailed(e)) => {
            CliOutput::error(&format!("finalize failed: {e}"));
            exit_code(&e)
        }
        Err(e) => {
            CliOutput::error(&format!("wizard error: {e}"));
            exit_code(&e)
        }
    }
}
```

**关键设计决策**：
- `init` 是**唯一退出后 eflow 完全退出的 subcommand**
- 退出码：0 = 成功 / 1 = 用户 Esc 取消 / 2 = 写文件失败

#### `stdin.rs` —— stdin 协议

```rust
// src/cli/stdin.rs
use tokio::io::{AsyncBufReadExt, BufReader};
use tokio::io::stdin;

#[derive(Deserialize)]
#[serde(tag = "action", rename_all = "snake_case")]
pub enum StdinCommand {
    Send { task_id: Option<Uuid>, task: String },
    End { task_id: Uuid },
    Level { task_id: Uuid, level: String },
    Lang { task_id: Option<Uuid>, locale: String },
    Help,
}

pub async fn read_loop(concierge: &mut Concierge) -> ExitCode {
    let stdin = stdin();
    let mut reader = BufReader::new(stdin);
    let mut line = String::new();
    
    loop {
        line.clear();
        match reader.read_line(&mut line).await {
            Ok(0) => return ExitCode::from(0),  // EOF
            Ok(_) => {
                let trimmed = line.trim();
                if trimmed.is_empty() {
                    continue;  // 空行跳过
                }
                
                match serde_json::from_str::<StdinCommand>(trimmed) {
                    Ok(StdinCommand::Send { task_id, task }) => {
                        dispatch_send(concierge, task_id, &task).await;
                    }
                    Ok(StdinCommand::End { task_id }) => {
                        dispatch_end(concierge, task_id).await;
                        return ExitCode::from(0);
                    }
                    Ok(StdinCommand::Level { task_id, level }) => {
                        dispatch_level(concierge, task_id, &level).await;
                    }
                    Ok(StdinCommand::Lang { task_id, locale }) => {
                        dispatch_lang(concierge, task_id, &locale).await;
                    }
                    Ok(StdinCommand::Help) => {
                        dispatch_help(concierge).await;
                    }
                    Err(e) => {
                        // 解析失败 → stderr 报错 + **继续**读下一行
                        CliOutput::error(&format!("stdin parse failed: {e}"));
                    }
                }
            }
            Err(e) => {
                CliOutput::error(&format!("stdin read error: {e}"));
                return ExitCode::from(2);
            }
        }
    }
}
```

**关键设计决策**：
- **`Send` 的 `task_id` 可选**——GUI 不必先知道 task_id；eflow 自动生成
- 用 `#[serde(tag = "action")]` 做 enum 标签——JSON 直观
- **解析失败不退出**——stdin 网络抖动时 GUI 偶尔发坏 JSON 不该让 eflow 死
- **EOF 退出码 0**（用户主动关闭 stdin 是正常退出）

#### 5 个 handler（`src/cli/handlers/`）

每个 handler 是 `SlashCommand` trait impl 的"薄包装"：

```rust
// src/cli/handlers/send.rs
pub async fn dispatch_send(concierge: &mut Concierge, task_id: Option<Uuid>, task: &str) {
    let final_task_id = task_id.unwrap_or_else(Uuid::new_v4);
    
    // 调 Concierge::handle_input（spec B1 已有 trait 集成）
    if let Err(e) = concierge.handle_input_with_id(final_task_id, task.to_string()).await {
        CliOutput::error(&format!("send failed: {e}"));
        return;
    }
    
    // 输出 ack
    let _ = CliOutput::json(&serde_json::json!({
        "ack": "send",
        "task_id": final_task_id,
        "dispatched_at": chrono::Utc::now().to_rfc3339(),
    }));
}

// src/cli/handlers/level.rs
pub async fn dispatch_level(concierge: &mut Concierge, task_id: Uuid, level: &str) {
    // 调 SlashCommand trait dispatch（spec B1 的 LevelCmd 空壳）
    let args = SlashArgs::from_kv(&[("task_id", &task_id.to_string()), ("level", level)]);
    match concierge.command_registry.dispatch(&format!("level {level}")) {
        Some((name, parsed_args)) => {
            let cmd = concierge.command_registry.get(name).unwrap();
            match cmd.execute(parsed_args, &mut CommandContext { ... }).await {
                Ok(SlashOutput::Text(s)) => {
                    let _ = CliOutput::json(&serde_json::json!({
                        "ack": "level",
                        "task_id": task_id,
                        "level": level,
                        "info": s,
                    }));
                }
                Ok(_) => {
                    let _ = CliOutput::json(&serde_json::json!({
                        "ack": "level",
                        "task_id": task_id,
                        "level": level,
                    }));
                }
                Err(e) => {
                    CliOutput::error(&format!("level failed: {e}"));
                }
            }
        }
        None => {
            CliOutput::error("level command not registered");
        }
    }
}

// src/cli/handlers/lang.rs - 类似 level
// src/cli/handlers/help.rs - 类似 level
// src/cli/handlers/end.rs - 调 concierge.shutdown() 不通过 trait
```

**关键设计决策**：
- **`send` 直接调 `Concierge::handle_input_with_id`**——不走 SlashCommand trait（task dispatch 是 Concierge 核心功能）
- **`end` 调 `Concierge::shutdown()`**——不走 SlashCommand trait（end 是 Concierge 生命周期管理）
- **`level` / `lang` / `help` 走 SlashCommand trait dispatch**——和 TUI 共用同一份 trait impl
- 每个 handler 成功后输出 ack JSON 到 stdout

## 4. 数据流 & 错误处理

### 4.1 数据流

#### 启动时（main.rs）

```
1. main.rs 解析 clap
2. match subcommand:
   None → run_tui_mode() (v1.3.1 行为零变化)
   Some(Init) → run_init() → Wizard finalize → exit 0/1/2
   Some(SessionStart) → run_session_start() → 持续运行（见下）
```

#### `eflow session start` 持续运行

```
1. 加载 config（~/.eflow/config.yaml 或 --config 指定）
2. 构造 LlmRouter（spec A）
3. 构造 CommandRegistry（spec B1，6 个命令注册 + required_register 校验）
4. 构造 Concierge（v1.2 + spec B1 trait 集成）
5. 订阅 Concierge event channel
6. 输出 SystemReady 事件到 stdout（**第一行**）
7. tokio::select! 并行跑 2 个任务：
   - 任务 A: 监听 event channel → 6 个事件 → CliOutput::ndjson_event()
   - 任务 B: 读 stdin → 解析 StdinCommand → dispatch 到 5 个 handler
8. Ctrl+C / EOF / panic 触发主循环退出
9. 退出时 exit_code(&err) (0/1/2/130)
```

#### GUI 套壳 Python 实际形态

```python
import subprocess, json

proc = subprocess.Popen(
    ["eflow", "session", "start", "--lang", "en-US"],
    stdin=subprocess.PIPE,
    stdout=subprocess.PIPE,
    stderr=subprocess.PIPE,
    text=True,
    bufsize=1,  # 行缓冲

# 读第一行 SystemReady（GUI 知道 eflow 启动完成）

    elif event["event_type"] == "TaskFailed":

# 写 end 指令
proc.stdin.write(json.dumps({"action": "end", "task_id": task_id}) + "\n")

proc.wait()
```

### 4.2 错误处理表

| 边界 | 谁负责 | 失败时 |
|---|---|---|
| `~/.eflow/config.yaml` 不存在 | main.rs | `eflow session start` 输出 exit code 1 + stderr 错误 |
| `~/.eflow/providers/` 不存在 + 无 env var | LlmRouter | exit code 2（系统错误） + stderr |
| stdin 收到非法 JSON | stdin.rs | stderr 输出错误 + **继续读下一行**（不退出） |
| stdin 收到合法 JSON 但 action 未知 | stdin.rs | stderr 输出错误 + 继续读下一行 |
| `send` 收到但 Concierge 已 shutdown | Concierge | stderr 输出错误 + 任务 A 继续跑 |
| `level` 收到非法 level 值（不是 simple/standard/advanced） | SlashCommand | exit code 1 + stderr |
| `lang` 收到不在 SUPPORTED_LOCALES 的 locale | SlashCommand | exit code 1 + stderr |
| `end` 收到不存在的 task_id | Concierge | stderr 警告 + 继续（不退出） |
| `Ctrl+C` | signal handler | exit code 130 + 关 stdin / 关 stdout |
| Concierge 内部 panic | task A | exit code 2 + 整个 eflow 进程退出 |
| start 启动时 home 目录不存在 | Config loader | exit code 1 + stderr |
| start 启动时 LlmRouter 初始化失败 | LlmRouter | exit code 2（系统错误） + stderr |
| stdout / stdin pipe 被 GUI 关闭 | OS / Rust runtime | exit code 1 + 整个 eflow 进程退出 |
| stderr 输出时 GUI 已不读 stderr | OS | Rust 忽略 EPIPE 错误（**不**让 eflow 死） |
| 多个 stdin `send` 同时到达 | Concierge | Concierge 内部串行处理（v1.2 既有）；事件按到达顺序输出到 stdout |
| `eflow init` 在向导 step 1 按 Esc | Wizard | exit code 1（用户取消） + 不写文件 |
| `eflow init` finalize 写文件失败 | Wizard | exit code 2（系统错误） + stderr |

### 4.3 错误处理原则

1. **stdout 永远 JSON 契约**——不掺杂任何人类可读字符
2. **stderr 永远人类可读**——GUI 不解析；用户调试时直接看
3. **stdin 解析失败不退出**——GUI 网络抖动时偶尔发坏 JSON 不让 eflow 死
4. **exit code 区分用户错误 vs 系统错误**——GUI 能决定是否重试
5. **EPIPE 不传播**——GUI 关 pipe 时 eflow 不该 panic；用 `let _ = writeln!(...)` 模式

## 9. ADR

### ADR-0016：TUI 仍是同进程 trait dispatch，subcommand 是 headless 包装

**状态**：已接受

**背景**：用户原话"GUI 套壳 CLI"——TUI 不能跨进程 spawn 自己；v4.0 §14.4 写的"InteractionLayer trait 同进程接入"在 TUI 实际工作中发现：(1) 单二进制部署与同进程绑定冲突；(2) 限制 GUI 技术栈为 Rust；(3) "核心层无感知"在 v1.2 实际未实现。

**决策**：v1.3.2 spec B2 起
- TUI 走 spec B1 的同进程 `CommandRegistry::dispatch()`（**零改造**）
- GUI 走 spec B2 的跨进程 CLI 契约（`eflow session start` + stdin/stdout 协议）
- 两条路径**共享同一份 `SlashCommand` trait impl**（spec B1 实施后已稳定）
- spec B2 **不"改造" TUI**，只加 subcommand 入口

**代价**：TUI 和 GUI 是两条不同入口路径，**测试要分别覆盖**；v1.3.2 任务量 +30%（多 3-4 个 task）

**收益**：
- 单二进制部署不变
- GUI 团队不依赖 eflow 编译
- TUI 和 GUI 行为 100% 一致（同一份 trait impl）
- 第三方可写 GUI 插件（任何语言）
- v2.0 GUI 实施工作量降低（直接 spawn vs 写 Rust 调 trait）

### ADR-0017：CLI 契约冻结 v1.3.0 起

**状态**：已接受

**背景**：v2.0 GUI 团队依赖契约稳定；GUI 一旦发布**不能轻易升级 eflow**（用户机器版本不一）。契约**只加不删**是 GUI 团队的硬要求。

**决策**：v1.3.0（spec A）起
- 6 个原有事件 + 1 个 `SystemReady` 的 schema **冻结**（合计 7 个）
- stdin 5 个 action 的 schema **冻结**
- exit code 4 档（0/1/2/130）**冻结**
- stdout 永远 JSON / stderr 永远人类可读 **冻结**
- 新增事件 / action / exit code 走 v1.4 候选 + CHANGELOG 标注
- **不**引入 `protocol_version` 字段（GUI 和 eflow 必须同版本部署）

**代价**：v1.4 加新事件要走 ADR；GUI 团队升级 eflow 时要看 CHANGELOG

**收益**：
- v2.0 GUI 一旦发布**长期可用**
- 契约有**单点权威**（`tests/gui_smoke_test.py` + `docs/cli-contract.md`）
- 6 + 1 个事件足够覆盖 99% 场景；新增需求走 v1.4 不打断 GUI

### ADR-0018：单 subcommand 模式（start 持续运行 + stdin 协议）

**状态**：已接受

**背景**：早期 spec B2 段 2 草稿设想"6 个独立 subcommand"——`eflow session start` / `send` / `events` / `end` / `level` / `lang`。实施时发现：eflow 是单进程单 Concierge 实体，**多 subcommand 进程无法共享内存**（daemon + IPC 是 v1.4+ 工作，需要 unix socket / HTTP / 共享内存）。

**决策**：v1.3.2 spec B2 起
- `eflow session start` **启动后持续运行**，stdout 输出 events NDJSON，stdin 接收指令
- 5 个 action（send/end/level/lang/help）通过 **stdin JSON 协议**实现
- **不**有独立的 `eflow session send` / `events` / `end` 等 subcommand
- GUI 套壳的实际形态 = "spawn eflow start，GUI 通过 stdin 发指令，stdout 收 events"

**代价**：GUI 必须 long-running 持有 eflow 子进程；stdin/stdout 是 GUI 唯一接口

**收益**：
- eflow 单进程架构不变
- 无 daemon / IPC 复杂度
- GUI 套壳代码极简（spawn + readline + writeline）
- v1.3 任务量减少 5-6 个 task（不需要 daemon + IPC 设计）
