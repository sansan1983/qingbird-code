## 0. 一句话定义

3 档工作流（Simple / Standard / Advanced）通过 `WorkflowExecutor` trait + `WorkflowRegistry` 注册，**核心零硬编码档位行为**。Concierge 在 dispatch 任务前**规则驱动**判定档位（零 LLM 成本），`/level` 命令提供**会话级 override**（重启清空，不持久化）。`/level` 委托给 spec B1 的 `SlashCommand` trait，spec C 只填具体逻辑。

---

## 1. 背景 & 动机

### 1.1 v1.3 用户硬性要求

用户在 v1.3 头脑风暴阶段提的 4 条硬性要求，第 4 条：

> 参考下 "forge" 软件的工作流 SKILL 借鉴 + 优化做到 工作流详细流程可 自动触发（A.简易工程流, B.标准工作流, C.高级系统化工作流），因为需要做到不同的项目任务适配不同的工作流。一个简单的几个 Step 就能完成的任务，用非常繁琐的工作流，反而降低了效率同时造成 token 的浪费。

**关键洞察**：

- 简单任务用复杂工作流 = **浪费 token**（用户原话）
- 复杂任务用简单工作流 = **质量不达标**
- 必须**自适应**档位——用户不需要每次都告诉 eflow "这次用简易"

### 1.2 v1.2 现状（债）

```
src/capability/
├── decisioner.rs               # 风险评估 + 模型路由（v1.0）
├── executor.rs                 # 步骤执行（v1.0）
└── feedbacker.rs               # 质量判决（v1.0）
```

**当前 `dispatch_task` 走固定流程**——Decisioner → Executor → Feedbacker 三角色管线，**没有"档位"概念**。所有任务一律 1 次反馈回路（`retry_count ≤ 1`），**不区分任务复杂度**。

### 1.3 spec B1 已铺垫

- `SlashCommand` trait + `CommandRegistry` 已定义（spec B1）
- `/level` 命令已注册为 `LevelCmd` impl，但**空壳只回 info message**："档位切换将在 v1.3.2 启用（spec C）"
- spec B1 文档显式标注"`/level` 语义留给 spec C"

### 1.4 设计哲学（核心原则）

**轻量化内核 + 高标准要求**（用户 2026-06-17 明确）：

- 核心 crate 不写死档位行为
- 加新档位 = 写 1 个 `impl WorkflowExecutor` + 1 行 `register()`，**核心零修改**
- **零 LLM 成本判定档位**——基于字符串匹配（关键词 + 长度 + 文件数）
- **override 优先于自动判定**——`/level` 设了就不调自动判定
- **override 会话级不持久化**——重启 eflow 后清空

## 2. 目标

### 2.1 核心目标

1. **`WorkflowLevel` enum**（3 个值：Simple / Standard / Advanced）
2. **`WorkflowExecutor` trait** + 3 个内置档位 impl
3. **`WorkflowRegistry`**（register / set_override / execute）
4. **Concierge 任务级档位判定**（5 条规则，按优先级匹配）
5. **`/level` 斜杠命令挂档位语义**（覆盖 spec B1 的空壳）
6. **会话级 `/level` override**（重启清空，不持久化）

### 2.3 显式不做

- **档位自动判定的 LLM 方案**（基于 LLM 判定档位会**反过来消耗 LLM**，违反零阻塞对话原则）—— 严格规则驱动
- **档位配置持久化**（`/level` 改的是会话级 state，不写回 eflow.yaml）—— 留给 v1.4 候选
- **档位 UI 显示**（TUI header 显示当前档位）—— **spec D 范围**（spec D 渲染引擎重构时一起做）
- **档内档位切换**（一个任务执行到一半切档）—— 任务级粒度，**不支持**档内切换
- **新档位（如 Turbo / Debug）**—— spec C 只做 3 个，**加新档位零改 core**（trait 注册机制）
- **档位相关的 system prompt 切换**（Simple 用一句话 prompt，Advanced 用复杂 prompt）—— 留给 v1.4 候选
- **档位相关的 LLM 路由**（Simple 强制用 light tier，Advanced 强制用 strong tier）—— 留给 v1.4 候选
- **档位相关的 risk threshold**（Advanced 自动 L3，Simple 自动 L0）—— 留给 v1.4 候选
- **Tab 补全 `/level` 参数**—— 留给 v1.4 候选

```
src/interaction/slash/builtin/
```

**关键设计决策**：

1. **`workflow/` 是独立目录**—— 3 个档位 impl 各 1 个文件，加新档位 = 加文件 + 1 行 `register()`，**核心零硬编码档位**
2. **`SimpleWorkflow` 跳过三角色管线**——直接调 1 次 LLM（ExpeditedAnswerPath）；`StandardWorkflow` 调三角色管线；`AdvancedWorkflow` 调三角色 + 记忆检索 + 多步规划 + 反馈 ×3
3. **`/level` 命令通过 `WorkflowRegistry::set_override(level)`**——会话级 override
4. **Concierge::dispatch_task 先调 `determine_workflow_level`，再调 `WorkflowRegistry::execute(level, task)`**

### 3.3 核心 trait

```rust
// src/workflow/mod.rs

#[derive(Debug, Clone, Copy, PartialEq, Eq, Deserialize, Serialize)]
#[serde(rename_all = "snake_case")]
pub enum WorkflowLevel {
    Simple,
    Standard,
    Advanced,
}

impl WorkflowLevel {
    pub fn label(&self) -> &'static str {
        match self {
            Self::Simple => "simple",
            Self::Standard => "standard",
            Self::Advanced => "advanced",
        }
    }
}

#[async_trait]
pub trait WorkflowExecutor: Send + Sync {
    fn level(&self) -> WorkflowLevel;
    fn description(&self) -> &'static str;  // /help 显示
    fn max_retries(&self) -> u8 { 3 }         // 默认 3，Advanced 可以重写
    async fn execute(&self, ctx: &mut WorkflowContext) -> Result<AggregatedResult, EflowError>;
}

pub struct WorkflowContext<'a> {
    pub task: &'a TaskSpec,
    pub concierge: &'a mut Concierge,
    pub orchestrator: &'a mut Orchestrator,
    pub memory: &'a mut MemoryManager,
}

pub struct WorkflowRegistry {
    executors: HashMap<WorkflowLevel, Arc<dyn WorkflowExecutor>>,
    /// 会话级 override（None = 走自动判定）
    override_level: Option<WorkflowLevel>,
}

impl WorkflowRegistry {
    pub fn new() -> Self;
    pub fn register(&mut self, exec: Arc<dyn WorkflowExecutor>);
    pub fn set_override(&mut self, level: Option<WorkflowLevel>);
    pub fn current_level(&self) -> WorkflowLevel {  // 当前生效的档位
        self.override_level.unwrap_or(/* 默认 Standard */ WorkflowLevel::Standard)
    }
    pub async fn execute(&self, ctx: &mut WorkflowContext) -> Result<AggregatedResult, EflowError>;
}
```

**关键设计决策**：
- `WorkflowRegistry::current_level()` **不调 LLM**——只是返回 override 或默认 Standard；**自动判定由 Concierge 在 dispatch 之前做**
- `execute` 委托给 `executors[level]`——**dispatch 路由**
- `set_override(None)` 清除 override（`/level auto` 切回自动判定）
- **3 个档位 trait impl 各自实现自己的 `execute`**

### 3.4 3 个档位的实现

#### `SimpleWorkflow`

```rust
// src/workflow/builtin/simple.rs
pub struct SimpleWorkflow;

#[async_trait]
impl WorkflowExecutor for SimpleWorkflow {
    fn level(&self) -> WorkflowLevel { WorkflowLevel::Simple }
    
    fn description(&self) -> &'static str {
        t!("workflow_simple_desc").as_str()  // "1 次 LLM 直接答，无验证"
    }
    
    async fn execute(&self, ctx: &mut WorkflowContext) -> Result<AggregatedResult, EflowError> {
        // 1. 构造 chat request（直接用 Concierge 的 router）
        let request = ChatRequest::new(
            ctx.task.workflow_default_model(),  // 用 light tier
            vec![Message::user(&ctx.task.description)],
        );
        
        // 2. 1 次 LLM 调用
        let response = ctx.concierge.router_mut().chat(ModelTier::Light, request).await?;
        
        // 3. 返回 AggregatedResult
        Ok(AggregatedResult {
            summary: response.content,
            // 无 steps / 无 feedback
            steps: vec![],
            feedback: vec![],
            workflow_level: WorkflowLevel::Simple,
        })
    }
}
```

**关键设计决策**：
- **不走三角色管线**——没有 Decisioner 评估，没有 Executor 工具调用，没有 Feedbacker 验证
- **强制 light tier**——Simple 不应该用大模型
- **无反馈回路**——错了用户自己改

#### `StandardWorkflow`

```rust
// src/workflow/builtin/standard.rs
pub struct StandardWorkflow;

#[async_trait]
impl WorkflowExecutor for StandardWorkflow {
    fn level(&self) -> WorkflowLevel { WorkflowLevel::Standard }
    
    fn description(&self) -> &'static str {
        t!("workflow_standard_desc").as_str()  // "三角色管线，1 次反馈回路"
    }
    
    async fn execute(&self, ctx: &mut WorkflowContext) -> Result<AggregatedResult, EflowError> {
        // 直接调 orchestrator（v1.0/v1.1 既有逻辑）
```

**关键设计决策**：
- **复用 v1.0/v1.1 的 orchestrator 逻辑**——零分叉
- 1 次反馈回路（feedback_log.len() ≤ 1）

#### `AdvancedWorkflow`

```rust
// src/workflow/builtin/advanced.rs
pub struct AdvancedWorkflow;

#[async_trait]
impl WorkflowExecutor for AdvancedWorkflow {
    fn level(&self) -> WorkflowLevel { WorkflowLevel::Advanced }
    
    fn description(&self) -> &'static str {
        t!("workflow_advanced_desc").as_str()  // "三角色 + 记忆 + 多步 + 3 次反馈"
    }
    
    fn max_retries(&self) -> u8 { 3 }  // 覆盖默认
    
    async fn execute(&self, ctx: &mut WorkflowContext) -> Result<AggregatedResult, EflowError> {
        // 1. 记忆检索（按 task 关键词）
        let memory = ctx.memory.recall(&ctx.task.description, RecallScope::Project, 5).await?;
        
        // 2. 注入到 Blackboard
        let blackboard = ctx.orchestrator.blackboard_mut();
        for m in memory {
            blackboard.scratchpad.insert(format!("memory_{}", m.id), serde_json::Value::String(m.content));
        }
        
        // 3. 调 orchestrator + 3 次反馈回路
        ctx.orchestrator.dispatch_with_max_retries(ctx.task, 3).await
    }
}
```

**关键设计决策**：
- **记忆检索前置**——v1.0 已有 `MemoryManager::recall`，Advanced 档**主动**调
- **3 次反馈回路**——v1.0 默认 3 次，Advanced 显式声明
- **强模型**（Decisioner 用 strong tier）

### 3.5 Concierge 改造（最小）

```rust
// src/application/concierge.rs 改
impl Concierge {
    pub async fn handle_input(&mut self, raw: String) -> Result<()> {
        if let Some(cmd) = raw.strip_prefix('/') {
            // 斜杠命令分发（spec B1 已有）
            ...
        } else {
            // 任务 dispatch：先判定档位
            let task = TaskSpec { description: raw, ..default() };
            let level = self.determine_workflow_level(&task);  // 规则驱动
            let effective_level = self.workflow_registry.override_level().unwrap_or(level);
            
            let task = TaskSpec { 
                workflow_level: effective_level,
                ..task 
            };
            
            self.dispatch_task(task).await
        }
    }
    
    /// 规则驱动的档位判定（5 条规则）
    pub fn determine_workflow_level(&self, task: &TaskSpec) -> WorkflowLevel {
        let desc = &task.description;
        let len = desc.chars().count();
        
        // 规则 1: 涉及多文件 → Advanced
        let file_count = count_file_extensions(desc);
        if file_count >= 3 { return WorkflowLevel::Advanced; }
        
        // 规则 2: 含关键词 → Advanced
        if contains_workflow_keyword(desc) { return WorkflowLevel::Advanced; }
        
        // 规则 3: 短任务 → Simple
        if len < 30 { return WorkflowLevel::Simple; }
        
        // 规则 4: 中等任务 → Standard
        if len < 100 { return WorkflowLevel::Standard; }
        
        // 规则 5: 长任务 → Advanced
        WorkflowLevel::Advanced
    }
}

fn count_file_extensions(s: &str) -> usize {
    ["rs", "py", "js", "ts", "go", "java", "cpp", "c", "h", "toml", "yaml", "yml", "json", "md"]
        .iter()
        .map(|ext| s.matches(&format!(".{}", ext)).count())
        .sum()
}

fn contains_workflow_keyword(s: &str) -> bool {
    ["重构", "系统", "全部", "梳理", "优化", "refactor", "refactoring", "system", "all", "optimize"]
        .iter()
        .any(|kw| s.contains(kw))
}
```

**关键设计决策**：
- **5 条规则按优先级匹配**——匹配到 1 条就返回
- **关键词检测支持中英**——`重构` / `系统` / `全部` / `refactor` / `system` / `refactoring`
- **文件扩展名 heuristic**——`.rs` / `.py` / `.js` / `.ts` / `.go` / `.java` / `.cpp` / `.c` / `.h` / `.toml` / `.yaml` / `.yml` / `.json` / `.md`
- **零 LLM 成本**——纯字符串操作
- **辅助函数独立可测**——`count_file_extensions` / `contains_workflow_keyword` 可单独单元测试

### 3.6 `/level` 命令挂档位语义

```rust
// src/interaction/slash/builtin/level.rs 改（spec B1 已是空壳）
pub struct LevelCmd;

#[async_trait]
impl SlashCommand for LevelCmd {
    fn name(&self) -> &'static str { "level" }
    fn help(&self) -> &'static str { "切工作流档位（simple/standard/advanced/auto）" }
    
    fn parse_args(&self, raw: &str) -> Result<SlashArgs, EflowError> {
        let level = raw.trim();
        match level {
            "simple" | "standard" | "advanced" | "auto" => Ok(SlashArgs::from_kv(&[("level", level)])),
            _ => Err(EflowError::Config(t!("err_invalid_level", level = level).to_string())),
        }
    }
    
    async fn execute(&self, args: SlashArgs, ctx: &mut CommandContext) -> Result<SlashOutput, EflowError> {
        let level = args.get("level").unwrap();
        match level {
            "auto" => ctx.concierge.workflow_registry_mut().set_override(None),
            "simple" => ctx.concierge.workflow_registry_mut().set_override(Some(WorkflowLevel::Simple)),
            "standard" => ctx.concierge.workflow_registry_mut().set_override(Some(WorkflowLevel::Standard)),
            "advanced" => ctx.concierge.workflow_registry_mut().set_override(Some(WorkflowLevel::Advanced)),
            _ => unreachable!(),
        }
        Ok(SlashOutput::Text(t!("status_level_changed", level = level).to_string()))
    }
}
```

**关键设计决策**：
- **`/level auto` 是新增**——切回"自动判定"（清除 override）—— spec B1 文档提的"3 档"扩展为 4 档（simple/standard/advanced/**auto**）
- **help 文案**说"切工作流档位"——不暴露实现细节（"override"等技术词不进 i18n）

## 4. 组件

spec C 的 5 个核心组件，每个独立可测。

### 4.1 C1. `WorkflowLevel` enum + `WorkflowExecutor` trait + `WorkflowRegistry`（`src/workflow/mod.rs`）

**单一职责**：档位抽象 + 注册表 + dispatch。**不做**档位实现，不做档位判定。

**规则**：
- `register` 加重复 level → 第二个赢 + warn
- `set_override(None)` 清除 override，`set_override(Some(L))` 设 override
- `current_level()` 返回 override 或默认 `Standard`——**不**调自动判定（自动判定由 Concierge 调）
- `execute(level, ctx)` 委托给 `executors[level]`——缺失 level → `Err(Internal("workflow level not registered"))`
- `required_register(&[levels])` 校验必需档位已注册，缺失 → 启动失败

**测试**：
- register + execute 成功
- register 同 level → 第二个赢
- execute 不存在的 level → Err
- `set_override` / `current_level` 切换
- `required_register` 缺失 → Err

### 4.2 C2. 3 个内置档位 impl（`src/workflow/builtin/*.rs`）

每个档位是独立 `impl WorkflowExecutor`。**关键设计决策**：

- **3 个 impl 各 1 个文件**——和 SlashCommand / WizardStep 同套哲学
- **`SimpleWorkflow.execute` 跳过 orchestrator**——直接 1 次 LLM 调用
- **`StandardWorkflow.execute` 调 orchestrator**——复用 v1.0/v1.1 既有逻辑
- **`AdvancedWorkflow.execute` 调 memory + orchestrator**——记忆检索 + 3 次反馈
- **每个 impl 都有 `description()`**——`/help workflow` 显示（v1.4 候选）

**测试**（每档位 2-3 个）：
- `SimpleWorkflow::execute` mock LLM → 1 次调用 + light tier + 无 steps
- `StandardWorkflow::execute` mock LLM → 三角色管线 + 1 次反馈
- `AdvancedWorkflow::execute` mock LLM + mock memory → 记忆检索 + 3 次反馈
- `description()` 非空且 i18n key 存在

### 4.3 C3. `Concierge::determine_workflow_level` 规则判定（`src/application/concierge.rs` 改）

**单一职责**：按任务字符串自动判定档位。**不做**档位执行。

**规则**（按优先级）：

| 规则 | 条件 | 判定结果 |
|---|---|---|
| 1 | 任务字符串含 ≥ 3 个文件扩展名（`.rs`/`.py`/`.js`/etc.） | Advanced |

## 5. 数据流 & 错误处理

### 5.1 数据流

#### 启动时（main.rs）

```
1. main.rs 构造 WorkflowRegistry
2. register 3 个 impl（Simple / Standard / Advanced）
3. required_register 校验 3 个全部注册
4. concierge.workflow_registry = workflow_registry
5. 启动完成
```

#### TUI 运行时（用户发任务）

```
1. 用户在 prompt 输入任务字符串 s，按 Enter
2. Concierge::handle_input(s):
   if s.starts_with('/'):
     # 斜杠命令（spec B1）
     dispatch(cmd) → SlashOutput
   else:
     # 任务 dispatch（spec C 改造）
     task = TaskSpec { description: s, ..default() }
     level = concierge.determine_workflow_level(&task)  # 规则驱动
     override = workflow_registry.override_level()
     effective = override.unwrap_or(level)
     task.workflow_level = effective
     
     # 推 workflow_dispatch_decision 到 messages（用户能看到判定结果）
     messages.push(t!("workflow_dispatch_decision", 
                      level = effective.label(), 
                      rule = rule_name(level)))
     
     # 派发到 workflow_registry
     let mut ctx = WorkflowContext { task, concierge, orchestrator, memory };
     let result = workflow_registry.execute(effective, &mut ctx).await?;
```

#### `/level` 命令运行时

```
1. 用户在 prompt 输入 "/level simple"，按 Enter
2. Concierge::handle_input("/level simple"):
   if s.starts_with('/'):
     dispatch("level simple") → LevelCmd::execute()
3. LevelCmd::parse_args("simple") → SlashArgs({level: "simple"})
4. LevelCmd::execute(args, ctx):
   workflow_registry.set_override(Some(Simple))
   return SlashOutput::Text("工作流档位已切换到 simple")
5. Concierge 推 status_level_changed 到 messages
```

### 5.2 关键边界

| 边界 | 谁负责 | 失败时 |
|---|---|---|
| 3 个档位 impl 未全部注册 | main.rs::required_register | 启动失败 + exit code 1 |
| `WorkflowLevel::parse` 收到非法字符串 | serde | `Err(serde_json::Error)` |
| `/level` 收到非法参数 | LevelCmd::parse_args | `Err(EflowError::Config(...))` 推到 messages |
| `/level` 调 set_override 时 workflow_registry 内部错误 | WorkflowRegistry | **不**应该发生（registry 是 in-memory） |
| 任务 dispatch 时 execute 出错 | WorkflowExecutor::execute | 推 t!("err_task_failed", msg) 到 messages |
| Simple 档 1 次 LLM 调用失败 | router.chat | 推 t!("err_llm_provider") 到 messages |
| Standard 档三角色管线失败 | orchestrator | 推 t!("err_task_failed", msg) 到 messages |
| Advanced 档记忆检索失败 | MemoryManager::recall | warn + 用空记忆继续（不阻塞） |
| Advanced 档 3 次反馈都失败 | orchestrator | 推 t!("err_task_failed_after_retries") 到 messages |
| `determine_workflow_level` 规则冲突（多文件 + 短任务） | 规则 1 优先 | 规则按顺序匹配，**不**做 OR |
| `override_level` 与 `current_level` 不一致 | WorkflowRegistry | **不**应该发生（set_override 原子操作） |
| `execute` 找不到 level（registry 被清空） | WorkflowRegistry | `Err(Internal("workflow level not registered"))` |

### 5.3 错误处理原则

1. **档位判定零 LLM 成本**——纯字符串匹配，**绝不**调 LLM 来判定档位
2. **override 优先于自动判定**——`/level` 设了就不调 `determine_workflow_level`
3. **override 会话级不持久化**——重启 eflow 后清空，**不**写回 eflow.yaml
4. **3 个档位 trait impl 必须全部注册**——`required_register` 启动时校验
5. **错误映射到现有 `EflowError` 变体**——不复用 `err_no_provider` 之类的 key，**用新 key**（档位是 spec C 新概念）
6. **EPIPE / GUI pipe 关闭不传播**——和 spec B2 一致

### 5.4 档位判定规则表（详细）

| 优先级 | 规则 | 判定 |
|---|---|---|
| 1 | 任务字符串含 ≥ 3 个文件扩展名 | Advanced |
| 2 | 含关键词（中文："重构" / "系统" / "全部" / "梳理" / "优化"） | Advanced |
| 2' | 含关键词（英文："refactor" / "refactoring" / "system" / "all" / "optimize"） | Advanced |
| 3 | 任务描述中文字数 < 30 | Simple |
| 3' | 任务描述英文字数 < 30 | Simple |
| 4 | 任务描述中文字数 < 100 | Standard |
| 4' | 任务描述英文字数 < 100 | Standard |
| 5 | 其他（中文字数 ≥ 100） | Advanced |

**关键设计决策**：
- **中英混合任务**按"任一关键词匹配"判定 Advanced
- **空任务**（空字符串）→ Simple（特殊兜底）
- **超长任务**（> 1000 字）→ Advanced
- **所有规则都用 `chars().count()`**——中文 1 字 = 1 字符，**不**用 byte length

## 10. ADR

### ADR-0019：核心零硬编码工作流档位

**状态**：已接受

**背景**：用户原话"轻量化内核 + 高标准要求"——3 个档位不能写死在 `match level` 里。`WorkflowExecutor` trait + `WorkflowRegistry` 注册表，**核心零硬编码档位行为**；加新档位 = 写 1 个 impl + 1 行 `register()`。

**决策**：v1.3.2 spec C 起
- `WorkflowLevel` enum（3 个值：Simple / Standard / Advanced）
- `WorkflowExecutor` trait + 3 个 impl（各 1 个文件）
- `WorkflowRegistry` 注册表 + required_register 启动校验
- Concierge 规则驱动的档位判定（5 条规则，零 LLM 成本）
- `/level` 命令提供会话级 override（重启清空）
- 核心代码零硬编码档位行为 / 判定规则 / override 语义

**代价**：每档位多写 1 个文件（30-50 行）；与 v1.0/v1.1 风格不同（v1.0 写死 3 角色管线）。

**收益**：
- 核心 crate 第三方依赖 ≤ 15 个（v4.0 铁律）
- 加新档位 = 1 个文件 + 1 行 `register()`，**零改 core**
- v1.4 加 "Turbo" / "Debug" 档零改 core
- GUI / TUI 共用同一份档位抽象
- 规则驱动判定零 LLM 成本
