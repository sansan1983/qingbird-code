# eflow v1.3.0 LLM 抽象扩展 · 设计文档

> **代号**：eflow v1.3.0 — 第一个 spec（LLM 抽象扩展）
> **日期**：2026-06-17
> **状态**：设计中，待用户审阅
> **作者**：头脑风暴对话
> **范围**：仅 LLM 抽象层扩展（spec A）。spec B（配置向导 + 多模交互 + CLI 契约）和 spec C（3 档工作流）独立成文

---

## 0. 一句话定义

把 v1.2 的"硬编码 2 家 provider"扩展为**配置驱动的 N 家 provider**，**核心 crate 零预置**。用户在 `~/.eflow/providers/{name}.yaml` 放一个元数据文件，eflow 启动时扫描注册。4 家预设厂商（DeepSeek / MiniMax / agnes-ai / OpenCode Go）以**示例 YAML 文件**形式存在，**不写进 core crate**。

---

## 1. 背景 & 动机

### 1.1 v1.2 现状（债）

v1.2 收尾时（17 commits ahead of origin/main，4 门禁全过）的 LLM 抽象层有 4 处硬编码：

1. `src/infrastructure/llm/anthropic.rs` — `AnthropicProvider` 具体 struct
2. `src/infrastructure/llm/openai.rs` — `OpenAiProvider` 具体 struct
3. `src/infrastructure/config.rs::ProvidersConfig` — `{anthropic: Option<...>, openai: Option<...>}` 写死 2 个字段
4. `src/infrastructure/llm/router.rs::LlmRouter::from_config` — `if let Some(ref anthro)` 硬编码分支

### 1.2 用户实际痛点

- 用户实际使用的 4 家 LLM（DeepSeek / MiniMax / agnes-ai / OpenCode Go）**没有一家是 Anthropic / OpenAI 直连**
- 扩展 1 家 LLM 必须改 4 处代码 + 重新编译 = **违反 eflow 核心 crate 轻量化原则**（v4.0 §1.3）
- v4.0 §2.1「核心策略」明确写"架构上保留完整能力规划，交付上按 feature flag 渐进式激活"——v1.2 的硬编码实现**和这个原则背道而驰**

### 1.3 触发本 spec 的 4 条硬性要求（用户题面）

1. 核心零预置 4 家厂商（仅示例 YAML）+ 自定义 provider 支持
2. 首次配置向导（spec B 范围，本 spec 仅做 `PresetLoader` 兜底）
3. 交互选择支持：输入序号 / ↑↓ / 鼠标滚轮（spec B 范围）
4. 3 档工作流自适应（spec C 范围）

**本 spec 只解决"1（仅抽象层）"**。spec B 和 spec C 独立成文。

## 2. 目标

### 2.1 核心目标

1. 用户创建 `~/.eflow/providers/deepseek.yaml` 后，启动 eflow 即可用 `deepseek-v4-pro` 模型，**核心 crate 零修改**
2. 用户创建 `~/.eflow/providers/my-proxy.yaml`（OpenAI 兼容，base_url 填自己）同样可用
3. v1.2 的 `ANTHROPIC_BASE_URL` / `OPENAI_BASE_URL` env var 行为**完全保留**（向后兼容）
4. 4 门禁全过：build / clippy -D warnings / fmt --check / cargo test

### 2.2 验收标准

| # | 验收项 | 测量方式 |
|---|---|---|
| A1 | `~/.eflow/providers/*.yaml` 自动加载 | 单元测试 + 手工 1 |
| A2 | OpenAI 兼容 / Anthropic 兼容 2 种协议都跑通 | 单元测试 + 手工 2 |
| A3 | model 列表端点（`GET /v1/models`）自动拉取 + 失败回退 | 单元测试（mock HTTP）|
| A4 | OpenCode Go "按 model 选 endpoint" 场景跑通 | 单元测试（断言 URL 拼接）|
| A5 | v1.2 env var 退化路径（providers 目录为空时） | 单元测试 + 手工 3 |
| A6 | 4 家 preset YAML 示例在 `docs/examples/providers/` | 文件存在检查 |
| A7 | 4 门禁全过 | CI / 本地 `cargo` 命令 |
| A8 | v1.2 的 9 个 anthropic/openai 测试全部继续通过 | 行为兼容 |

### 2.3 显式不做

- **不做 GUI 选择 / 不做 TUI 多模交互**（spec B）
- **不做工作流档位**（spec C）
- **不做 preset 厂商的二进制预置**（即使 4 家也是 YAML 文件，不写进 core）
- **不做 Anthropic 鉴权以外的协议扩展**（不支持 google / cohere / 自家协议；要 Google Gemini 走 OpenAI 兼容代理即可）
- **不引入新第三方 crate**（`serde_yaml` / `serde_json` / `reqwest` 全部已存在；`expand_env_vars` 内部用的 `regex_lite` 是 v1.2 已有的）
- **不破坏 `LlmProvider` trait 签名**（spec B 引入 CLI 契约时需要稳定抽象）

### 3.2 配置 schema

#### 3.2.1 `eflow.yaml` 顶层（破坏性变更）

**v1.2 形态**（废弃）：
```yaml
llm:
  providers:
    anthropic:
      api_key: ${ANTHROPIC_API_KEY}
      default_model: claude-sonnet-4-6
    openai:
      api_key: ${OPENAI_API_KEY}
      default_model: gpt-4o
  routing:
    strong: anthropic
    medium: anthropic
    light: openai
  cache:
    l1_enabled: true
    l2_enabled: false
    l2_ttl_days: 7
```

**v1.3 形态**（新增）：
```yaml
llm:
  routing:                       # 3 个 tier 各引用一个 provider id
    strong: "deepseek"           # Decisioner 用
    medium: "deepseek"           # Feedbacker 用
    light: "minimax"             # Executor 用
  cache:
    l1_enabled: true
    l2_enabled: false
    l2_ttl_days: 7
```

**变更点**：`providers.{anthropic, openai}` 字段**完全删除**。`routing.{strong, medium, light}` 不变（已引用 provider id）。

#### 3.2.2 `~/.eflow/providers/deepseek.yaml`（新增 schema）

```yaml
id: deepseek                    # 唯一 id，与 routing 引用对齐
display_name: "DeepSeek"        # TUI/log 显示用
protocol: openai_compatible     # openai_compatible | anthropic_compatible
base_url: "https://api.deepseek.com"
api_key: "${DEEPSEEK_API_KEY}"
default_model: "deepseek-v4-pro"
timeout_secs: 30
max_retries: 3
retry_backoff_ms: 1000
preset_models:                  # 拉取失败时的 fallback
  - "deepseek-v4-pro"
  - "deepseek-v4-flash"
list_models_endpoint: "/models"  # 拉取模型列表的路径（可选）
list_models:                     # OpenCode Go 特殊场景：按 model 选 endpoint
  - id: "glm-5.1"
    endpoint: "/v1/chat/completions"
  - id: "kimi-k2.7"
    endpoint: "/v1/messages"
extra_config: {}                # 兜底字段，generic adapter 不识别就忽略
```

**关键设计决策**：
- `list_models` 数组里的 `endpoint` 字段**专门解决 OpenCode Go 的"同 base URL 不同 endpoint"**——其他 3 家只填 string 列表或省略
- `extra_config` 是 v1.3 以后的扩展点，新字段优先加在这里 → 旧 YAML 不破坏
- 4 家 preset 文件**不进 core crate**，放在 `docs/examples/providers/` 当 sample

### 3.3 4 家 preset 厂商元数据表

| 文件 | protocol | base_url | preset_models |
|---|---|---|---|
| `deepseek.yaml` | openai_compatible | `https://api.deepseek.com` | `deepseek-v4-pro`, `deepseek-v4-flash` |
| `minimax.yaml` | openai_compatible | `https://api.minimaxi.com/v1` | `MiniMax-M3`, `MiniMax-M2.7`, `MiniMax-M2.5`, `MiniMax-M2.1`, `MiniMax-M2` |
| `agnes-ai.yaml` | openai_compatible | `https://apihub.agnes-ai.com/v1` | `agnes-2.0-flash`（仅 1 个） |
| `opencode-go.yaml` | openai_compatible | `https://opencode.ai/zen/go/v1` | 13 个模型，每个带 `endpoint` 字段分流 |

**这 4 个文件**不写进 `eflow-core`，用户在 `eflow init` 时（spec B）复制到 `~/.eflow/providers/`。

### 3.4 向后兼容（env var 退化路径）

`ANTHROPIC_BASE_URL` / `OPENAI_BASE_URL` env var **完全保留**：
- `~/.eflow/providers/` 目录**不存在** 或 **存在但无文件** → 自动 fallback 到 v1.2 行为
- 退化实现：复用 v1.2 的 `if let Some(anthro) = env::var(...)` 逻辑，包成 `fallback_from_env_vars() -> HashMap<String, Arc<dyn LlmProvider>>`
- 用户在 eflow.yaml 的 `routing.strong/medium/light` 仍引用 `anthropic` / `openai` 这两个内置 id

**接受破坏性变更**：v1.2 eflow.yaml 含 `providers.anthropic/openai` 字段时，serde 反序列化会**报错**。CHANGELOG 写明，迁移指南见 `docs/migration-v1.2-to-v1.3.md`。

## 4. 组件

5 个核心组件，每个独立可测。

### 4.1 C1. `ProviderConfig` 数据结构（`src/infrastructure/llm/types.rs` 新增）

```rust
#[derive(Debug, Clone, Deserialize)]
pub struct ProviderConfig {
    pub id: String,
    pub display_name: String,
    pub protocol: ProtocolKind,
    pub base_url: String,
    pub api_key: String,
    pub default_model: String,
    #[serde(default = "default_timeout_secs")]
    pub timeout_secs: u64,
    #[serde(default = "default_max_retries")]
    pub max_retries: u8,
    #[serde(default = "default_retry_backoff_ms")]
    pub retry_backoff_ms: u64,
    #[serde(default)]
    pub preset_models: Vec<String>,
    #[serde(default)]
    pub list_models_endpoint: Option<String>,
    #[serde(default)]
    pub list_models: Vec<ModelEntry>,
    #[serde(default)]
    pub extra_config: serde_json::Value,
}

#[derive(Debug, Clone, Copy, Deserialize, PartialEq, Eq)]
#[serde(rename_all = "snake_case")]
pub enum ProtocolKind {
    OpenaiCompatible,
    AnthropicCompatible,
}

#[derive(Debug, Clone, Deserialize)]
pub struct ModelEntry {
    pub id: String,
    #[serde(default)]
    pub endpoint: Option<String>,
}
```

**单一职责**：把 YAML 解析成强类型。**不做** I/O / HTTP / registry 逻辑。

### 4.2 C2. `PresetLoader` 扫目录（`src/infrastructure/llm/preset_loader.rs` 新增）

```rust
pub struct PresetLoader;

impl PresetLoader {
    /// 扫描 `~/.eflow/providers/*.yaml`，返回所有 ProviderConfig
    pub fn load_all(dir: &Path) -> Result<Vec<ProviderConfig>>;

    /// 单文件加载（测试用）
    pub fn load_one(path: &Path) -> Result<ProviderConfig>;
}
```

**规则**：
- 目录不存在 → 返回空 Vec（不是错误）
- `load_all` 时文件解析失败 → 跳过该文件 + `tracing::warn!`，不中断整体加载
- `load_one` 时文件解析失败 → 返回 `Err`（不 warn 不跳过，调用方必须处理）
- **id 重复** → 保留第一个 + warn（不 panic）
- YAML 展开 `${ENV_VAR}` 复用 `common::env::expand_env_vars`

**测试覆盖**：
- 空目录 → `Ok(vec![])`
- 单文件合法 → `Ok(vec![ProviderConfig])`
- 单文件 YAML 错误 → 跳过 + warn
- 两个文件同 id → 第一个赢 + warn
- `${EFLOW_TEST_VAR}` 展开成功

### 4.3 C3. `LlmProviderRegistry` 工厂（`src/infrastructure/llm/registry.rs` 新增）

```rust
pub struct LlmProviderRegistry;

impl LlmProviderRegistry {
    /// 把 ProviderConfig 列表转成 HashMap<id, Arc<dyn LlmProvider>>
    pub fn build(presets: Vec<ProviderConfig>) -> Result<HashMap<String, Arc<dyn LlmProvider>>>;
}
```

**规则**：
- `protocol: OpenaiCompatible` → 构造 `GenericOpenAiProvider`
- `protocol: AnthropicCompatible` → 构造 `GenericAnthropicProvider`
- **ListModels 时**走 `list_models_endpoint`（如果有）→ 拼 `base_url + endpoint` → HTTP GET
  - 成功 → 解析 JSON 数组 → 缓存到 `~/.eflow/cache/models/{provider_id}.json`（TTL 24h）
  - 失败 / 端点未配置 → 用 `preset_models`
  - **OpenCode Go 特殊场景**：`list_models` 数组里的 `endpoint` 字段在 chat 时被 generic provider 读取（"model X 用 endpoint Y"），不在 registry 阶段处理

**测试覆盖**：
- 单 preset 构造成功
- 两种 protocol 都跑通
- `list_models` 数组构造成功（不查 HTTP）
- 解析失败时返回原始错误

### 4.4 C4. `GenericOpenAiProvider` / `GenericAnthropicProvider`（改造）

从 v1.2 的 `OpenAiProvider` / `AnthropicProvider` 提取，**核心变化**：

| 旧 | 新 |
|---|---|
| `api_key: String`（构造参数） | 同 |
| `default_model: String` | 同 |
| `client: Client`（注入 timeout） | 同 |
| `base_url: String`（env var 注入） | 同（但不再只是"覆盖"——它是构造时唯一来源） |
| — | **`model_endpoints: HashMap<String, String>`**（OpenCode Go 场景，model id → endpoint path） |

**`chat_url` 改造点**：
```rust
fn chat_url(&self, model: &str) -> String {
    let path = self.model_endpoints
        .get(model)                              // OpenCode Go 优先按 model 找
        .map(String::as_str)
        .unwrap_or(self.default_chat_path);      // 否则用协议默认（OpenAI: /chat/completions, Anthropic: /v1/messages）
    format!("{}{}", self.base_url.trim_end_matches('/'), path)
}
```

**`build_body` 几乎不变**——已经是协议正确实现。

**`chat` / `chat_stream` 不变**。

**`name()` 改造**：从硬编码 `"openai"` / `"anthropic"` 改为返回 `self.config.id.as_str()`（router log 里看到的是"deepseek"而不是"openai"）。

**`supports_prefix_cache` 改造**：从硬编码 `true` 改为读 `extra_config.supports_prefix_cache`（默认 `true`，4 家都支持）。

**测试覆盖**：
- 复用 v1.2 现有 9 个测试（timeout / cache_control / base_url / messages_url / chat_url 拼接），断言改为新构造器
- 新增：`model_endpoints` 命中 → URL 用 model 专属 path
- 新增：`model_endpoints` miss → 回退到默认 path
- 新增：`name()` 返回 `config.id`

### 4.5 C5. `LlmRouter` 改造（`src/infrastructure/llm/router.rs` 改）

**`from_config` 完全重写**：

```rust
pub fn from_config(
    config: &EflowConfig,
    provider_dir: &Path,                          // 显式传 ~/.eflow/providers/
) -> Result<Self> {
    let presets = PresetLoader::load_all(provider_dir)?;
    let mut providers = LlmProviderRegistry::build(presets)?;

    // 退化路径：providers 为空时用 v1.2 的 env var 行为
    if providers.is_empty() {
        providers = fallback_from_env_vars()?;     // 复用 v1.2 的 if let Some(anthro) 逻辑
    }

    if providers.is_empty() {
        return Err(EflowError::Config(t!("err_no_llm_providers").to_string()));
    }

    let routing = build_routing(&config.llm.routing, &providers);
    // ... L2 cache 不变 ...
}
```

**`routing` 验证**：3 个 tier 引用的 provider id 必须都在 `providers` 字典里；缺失 → warn + 降级到第一个可用 provider（v1.2 行为保留）。

**测试覆盖**：
- 复用 v1.2 现有 4 个 router 测试（重试 / 降级 / L2 缓存）
- 新增：`provider_dir` 为空 → 退化到 env var
- 新增：`provider_dir` 有 provider → 不读 env var
- 新增：routing 引用不存在的 id → warn + 用第一个

### 4.6 公共：`expand_env_vars` 提取到 `src/common/env.rs`

```rust
/// 展开 ${ENV_VAR} 形式的字符串
pub fn expand_env_vars(input: &str) -> String;
```

`config.rs` 和 `preset_loader.rs` 都调用这个**唯一实现**。

### 4.7 稳定性约束（spec B CLI 契约的前置条件）

**为 v1.3 spec B 的 CLI 稳定契约铺路**，spec A 实施时保证：

1. `LlmProvider` trait 的方法签名在 v1.3 之后**不再破坏性变更** —— GUI 套壳时如果 LLM 调用变体，CLI 契约层会 panic
2. `ProviderConfig` schema 在 v1.3.0 后**只做 additive 变更**（加新字段不动旧字段）—— GUI 的 `eflow init` 会用同一份 schema
3. `~/.eflow/providers/*.yaml` 的位置、命名、glob 模式**不依赖具体 OS 路径**——GUI 套壳时也得能扫到

**这 3 条作为 spec A 的"非功能需求"**，实施时多花 1 个 task（主要是测试覆盖）。

## 5. 数据流 & 错误处理

### 5.1 数据流

**启动时**（`main.rs` → `LlmRouter::from_config`）：

```
1. main.rs 拿到 EflowConfig + 用户配置目录 (~/.eflow/)
2. main.rs 调用 LlmRouter::from_config(&config, &user_dir.join("providers"))
3. PresetLoader::load_all(provider_dir)
   ├── 扫 *.yaml → 逐个 serde_yaml::from_str
   ├── 每条 expand_env_vars(api_key)
   └── 返回 Vec<ProviderConfig>
4. LlmProviderRegistry::build(presets)
   ├── 逐个 ProviderConfig 按 protocol 构造 GenericXxxProvider
   ├── 每个 provider 内部：可选 HTTP GET list_models_endpoint
   │   ├── 成功 → 解析后存到 ~/.eflow/cache/models/{id}.json (TTL 24h)
   │   └── 失败 → 用 preset_models
   └── 返回 HashMap<id, Arc<dyn LlmProvider>>
5. LlmRouter::from_config 验证 routing → 构造 LlmRouter
```

**运行时**（`Concierge` → `Decisioner` → `LlmRouter::chat(tier, request)`）：

```
1. Decisioner 调 LlmRouter::chat(Strong, request)
2. Router 查 routing.strong → provider_id
3. 查 self.providers[provider_id] → Arc<dyn LlmProvider>
4. provider.chat_url(model)  →  按 model_endpoints 选 path → 拼 base_url
5. provider.chat(request)    →  HTTP POST
6. 返回 ChatResponse
```

### 5.2 错误处理表

| 边界 | 谁负责 | 失败时 |
|---|---|---|
| YAML 文件不存在 | PresetLoader::load_all | 静默返回空 Vec |
| YAML 解析失败（load_all 时） | PresetLoader::load_all | 跳过 + warn，不中断 |
| YAML 解析失败（load_one 时） | PresetLoader::load_one | 返回 `Err`，调用方处理 |
| `${ENV_VAR}` 未设置 | expand_env_vars | 留原样 `${VAR}`，provider 构造时 `api_key.is_empty()` 触发 v1.2 的"占位 key"检查 |
| provider id 重复 | PresetLoader | 第一个赢 + warn |
| `list_models_endpoint` HTTP 失败 | LlmProviderRegistry | 用 `preset_models` + warn |
| `list_models_endpoint` 返回 401 | LlmProviderRegistry | 同上（不区分错误码） |
| `list_models` 数组含 `endpoint: null` | GenericXxxProvider | 用协议默认 path |
| `routing.strong` 引用不存在的 id | LlmRouter::from_config | warn + 降级到 `providers.keys().next()` |
| 所有 provider 构造失败 | LlmRouter::from_config | 返回 `EflowError::Config("err_no_llm_providers")` |
| HTTP 调用时 api_key 无效 | provider.chat | 返回 `EflowError::LlmAuthFailed` |
| HTTP 429 限流 | LlmRouter::chat | 触发重试 + 降级（v1.2 行为） |

### 5.3 错误处理原则

1. **配置阶段尽量宽容** —— YAML 错、env var 缺、id 重复 → 跳过/warn，不让用户启动不了
2. **运行阶段严格** —— api_key 无效、HTTP 错误 → 返回明确错误，让用户重试
3. **退化路径必须工作** —— `~/.eflow/providers/` 为空 + env var 存在 → 完全复用 v1.2 行为

## 7. 风险 & 缓解

| 风险 | 可能性 | 影响 | 缓解 |
|---|---|---|---|
| **routing 引用 id 不存在 → 静默降级**导致用户配错不自知 | 中 | 中 | 启动时如果出现降级，**额外打一条 `tracing::warn!` 到 stderr** + 在 TUI 状态栏显示"⚠ routing 降级"红字 |

| **YAML 展开 `${VAR}` 和 `config.rs` 的 `expand_env_vars` 行为分裂** | 低 | 低 | 抽到 `src/common/env.rs` 作为唯一实现，两处都调它 |

| **v1.2 的 9 个 anthropic.rs / openai.rs 测试改构造器时**逻辑错误 | 中 | 中 | 行为 100% 兼容：base_url / api_key / model / timeout / cache_control / messages_url 拼接——这些在新构造器下**逻辑不变**，仅数据来源从硬编码 env var 变成显式 ProviderConfig |
| **核心 crate 第三方依赖增加**（设计文档铁律：≤ 15 个） | 低 | 中 | spec A **不引入任何新 crate** |
| **spec B 的 CLI 契约要求 LlmProvider trait 稳定，但本 spec 改动 trait 周边** | 低 | 高 | trait 本身**不动**；本 spec 只动具体实现（`name()` 签名不变，只改实现） |

## 9. 对设计文档的影响

### 9.1 v4.0 §2.1「核心策略」兑现

### 9.2 v4.0 §11.1「ModelTier 注释」修订

§11.1 写：
```rust
pub enum ModelTier {
    Strong,    // Decisioner: Claude Opus / GPT-4
    Medium,    // Feedbacker: Claude Sonnet / GPT-4o
    Light,     // Executor:   Claude Haiku / GPT-4o-mini
}
```

**修订建议**：注释改为中性

pub enum ModelTier {
    Strong,    // Decisioner: 路由到 routing.strong 配置的 provider

    Light,     // Executor:   路由到 routing.light 配置的 provider

## 10. ADR（架构决策记录）

### ADR-0011：核心 crate 零预置 provider

**状态**：已接受

**背景**：v1.0/v1.2 在 `src/infrastructure/llm/` 硬编码 Anthropic + OpenAI 两个具体类，`EflowConfig.ProvidersConfig` 也硬编码两个字段。扩展 1 家 LLM 必须改 4 处代码 + 重新编译，违反 eflow 核心 crate 轻量化原则（v4.0 §1.3）和"渐进式激活"原则（v4.0 §2.1）。

**决策**：v1.3 起
- `eflow-core` 不预置任何具体 provider
- 暴露 `GenericOpenAiProvider` / `GenericAnthropicProvider` 两个**通用协议 adapter**
- 用户在 `~/.eflow/providers/{name}.yaml` 放 ProviderConfig 元数据
- `LlmProviderRegistry` 扫目录构造 provider
- 4 家 preset 厂商以**示例 YAML** 形式存在（`docs/examples/providers/`），不进 core

**代价**：v1.3 比 v1.2 多 5-6 个 tasks 的 plugin 注册机制。

**收益**：
- 核心 crate 第三方依赖 ≤ 15 个（v4.0 铁律）
- 加 1 家 LLM = 复制 1 个 YAML 文件，零编译
- 第三方可发布 preset provider
- v4.0 §1.3 轻量化原则兑现

### ADR-0012：v1.2 → v1.3 配置格式破坏性变更

**状态**：已接受

**背景**：`EflowConfig.ProvidersConfig` 硬编码 `{anthropic, openai}` 字段在 v1.3 必须改为更通用的 `Vec<ProviderConfig>` 形式。保持向后兼容的代价（双 schema 解析层）比一次性破坏变更更大。

**决策**：v1.3.0 起
- `EflowConfig.ProvidersConfig` 字段完全删除
- v1.2 eflow.yaml 含 `providers.anthropic/openai` 时 serde 反序列化报错
- CHANGELOG 标注 "Breaking: v1.2 → v1.3 升级需迁移"
- 提供 `docs/migration-v1.2-to-v1.3.md` 迁移指南
- 提供 `docs/examples/providers/anthropic.yaml` / `openai.yaml` 样例

**代价**：用户必须手动迁移配置（1 个 yaml 字段复制动作）。

**收益**：
- 不维护双 schema 解析层
- 1 个统一的 schema 长期维护成本更低
- 配合 ADR-0011 彻底解决 LLM 抽象债

### ADR-0013：GUI 接入路径从 InteractionLayer trait 改为 CLI 稳定契约

**状态**：已接受

**背景**：v4.0 §14.4 设计 GUI 实现 InteractionLayer trait 同进程接入。实施时发现：
1. 单二进制部署与同进程绑定冲突（GUI 必须链 eflow 整个 crate = 编译产物巨大）
2. 限制 GUI 技术栈为 Rust（egui/iced）
3. v1.2 TUI 实际"核心层无感知"是错的（必须显式 `Arc<Mutex<Concierge>>` 传引用）
4. 用户提的"GUI 套壳 CLI 模式"是更彻底的解耦

**决策**：v1.3 spec B 实施时
- 定义"CLI 稳定契约"：`eflow session {start, send, events, end, level, lang}` 等子命令
- GUI 套壳用 stdio + events 通道跨进程接入
- InteractionLayer trait 保留给 TUI（仍同进程），但 TUI 内部走 CLI 契约自调用
- v2.0 GUI 直接 spawn eflow 进程，技术栈自由

**代价**：spec B 实施工作量 +30%（多 3-4 个 task）。

**收益**：
- 单二进制部署不变
- GUI 团队不依赖 eflow 编译
- TUI 和 GUI 行为 100% 一致（同一份契约）
- 第三方可写 GUI 插件
- v2.0 GUI 实施工作量降低（直接套壳 vs 写 Rust 调 trait）
