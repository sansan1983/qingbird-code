## 二、设计目标

1. **把"声明 ship 但实际未工作"的 14 个半成品全部接上** —— v0.2.18 + v0.2.19
2. **把 8 个日常可用打磨项交付** —— v0.3.0
3. **让 `qingbird --interactive` 成为日常编码助手** —— 终端用户的"v0.3.0 可用"判断标准
4. **架构上不欠新债** —— 删 14 个 write-only 类型、聚合错误、RuntimeOverrides 隔离瞬时态
5. **每个版本可独立发布** —— v0.2.18 / v0.2.19 任一发布失败不影响后续

### 3.4 Streaming(Q3 方案 4 + Q10 方案 B)

| 维度 | 决策 |
|---|---|
| 覆盖 provider | **5 个全做**(DeepSeek / OpenAI / Ollama 走 OpenAI SSE;Anthropic / DeepSeek-Anthropic 走 Anthropic SSE) |
| 触发方式 | `--stream` / `--no-stream` 显式 flag,默认关闭;**Profile 可覆盖默认** |
| 架构 | `enum StreamFormat { OpenAiCompatible, Anthropic }` + 共享 SSE parser(provider 只声明格式) |
| 交互 | content 增量直接 stdout(打字机);tool_call 增量缓冲,`Done` 后批量执行,再开新流 |
| 中断 | Ctrl+C → 丢当前流,已收 tool_call 一律丢弃 |
| 流式回退 | provider 的 stream() 默认实现收到不返 SSE 的响应 → 自动 fallback 非流式 + `tracing::warn!` |
| Token 精度 | 流式响应的 `usage` chunk(末尾)继续走 `UsageStats`,不丢精度 |
| i18n | 不需要新 key |

```rust
pub enum StreamFormat { OpenAiCompatible, Anthropic }

pub enum StreamEvent {

    ToolCallDelta { id: String, name: String, args_fragment: String },
    ToolCallComplete(ToolCall),
    Usage(UsageStats),
    Done,
    Error(String),
}

pub async fn parse_sse(
    body: impl AsyncRead, format: StreamFormat
) -> impl Stream<Item = StreamEvent>;
```

```rust
// providers/mod.rs
pub trait Provider {

    fn stream_format(&self) -> StreamFormat { StreamFormat::OpenAiCompatible }
```

### 4.1 Streaming 架构(方案 B)

**Provider trait 改动(向后兼容):**

```rust
// providers/mod.rs
#[async_trait]
pub trait Provider: Send + Sync {
    fn kind(&self) -> &str;
    fn protocol(&self) -> Protocol;
    fn model(&self) -> &str;
    fn base_url(&self) -> &str;
    fn endpoint(&self) -> &str { "/v1/chat/completions" }
    fn stream_format(&self) -> StreamFormat { StreamFormat::OpenAiCompatible }  // new method
    fn build_request_body(&self, ...) -> serde_json::Value;
    fn build_headers(&self, ...) -> Vec<(String, String)>;
    fn parse_response(&self, body: &serde_json::Value) -> Result<ChatResponse>;
    async fn send(&self, ...) -> Result<ChatResponse>;
    async fn stream(&self, req: &RequestConfig) -> Result<Pin<Box<dyn Stream<Item = Result<StreamEvent>> + Send>>> {
        // default impl: open stream_format's parser
        // on failure: fallback to send() + synthesize single StreamEvent::Done
    }
}
```

**react_loop 集成:**

```rust
// react_loop/mod.rs:流式分支
let stream = provider.stream(&req_config).await?;
let mut acc_text = String::new();
let mut tool_buffer = ToolCallBuffer::new();
let mut collected_tools: Vec<ToolCall> = Vec::new();
let mut usage = UsageStats::default();

while let Some(event) = stream.next().await {
    match event? {
        StreamEvent::TextDelta(s) => {
            if interactive_mode { print!("{s}"); stdout.flush()?; }
            acc_text.push_str(&s);
        }
        StreamEvent::ToolCallDelta { id, name, args_fragment } => {
            tool_buffer.accumulate(id, name, args_fragment);
        }
        StreamEvent::ToolCallComplete(tc) => collected_tools.push(tc),
        StreamEvent::Usage(u) => usage = u,
        StreamEvent::Done => break,
        StreamEvent::Error(e) => return Err(EflowError::Stream(e)),
    }
}

// stream done, synthesize ChatResponse
let response = ChatResponse {
    content: acc_text,
    tool_calls: collected_tools,
    finish_reason: "stop".into(),
    usage,
};
// fall through to the same Step transition logic as non-streaming path
```

```rust
async fn recall_memories(&self, query: &str) -> Vec<MemoryEntry> {
    if let Some(mm) = &self.memory_manager {
        match mm.search(query, Scope::Project).await {
            Ok(results) => results.into_iter().take(5).collect(),
            Err(e) => {
                tracing::warn!("memory recall failed: {e}");
                Vec::new()
            }
        }
    } else {
        Vec::new()
    }
}

async fn save_memory_async(&self, entry: MemoryEntry) {
    if let Some(mm) = self.memory_manager.clone() {
        tokio::spawn(async move {
            if let Err(e) = mm.save(entry).await {
                tracing::warn!("memory save failed: {e}");
            }
        });
    }
}
```

### 6.1 方向 C:Subagent 可组合(目标 v0.4 / v1.0)

**核心问题:** 当前 `Subagent::run` 是孤立的 `ReactLoop` 调用,无父↔子消息回流,无共享状态。

**关键模块:**

| EventChannel 接入 | react_loop 每次 Step 转换发 event,UI / 日志 / 未来 telemetry 订阅 |
| 任务规划模型 | v0.2.18 删掉的 `TaskSpec` / `PlannedStep` / `Intent` 类型,如果方向 C 需要,**重新设计**为"以 Subagent 为粒度" |

**预期里程碑:** v0.4.0 恢复 Pool + 父↔子消息;v0.5.0 EventChannel 接入;v1.0 多 agent 协作完整。

## 八、假设与未决项

### 假设(Implementation 期间发现不符需重新对齐)

1. **sqlite + bundled 编译** 在 Windows / macOS / Linux 三平台 CI 都能跑通(目前 CI 是 Linux-only,需要补矩阵)
2. **`similar = "2"`** 的 license 与现有 dep 兼容(MIT,应该没问题)
3. **DeepSeek-Anthropic provider 的 SSE 格式** 与官方 Anthropic 完全一致(否则需要单独 parser)
4. **Profile YAML 解析** 不需要支持 `!include` / 环境变量展开(如果需要,延后到 v0.4+)
5. **Anthropic 流式响应 `usage` 跟踪**:`input_tokens` 在 `message_start`,`output_tokens` 在 `message_delta`,需要合并两端;不是简单"末尾取一次"

### 未决项(留待 v0.2.18 实施时确认)

1. **默认 profile** 是否在首次安装时自动创建 `developer.yaml`?——倾向**不创建**,留给用户主动 `qingbird --profile init` 或文档引导
2. **`/session delete` 是否要 batch 删除**(`/session delete --before 2026-01-01`)?——v0.3.0 不做,v0.4+ 再议
3. **cost 字段是否要给汇率硬编码一个比 7.2 更合理的值**(OpenExchangeRates 等)?——v0.3.0 固定 7.2,标注是"近似值"
4. **Streaming 失败回退** 在 `tracing::warn!` 之外要不要打 `eprintln`?——倾向仅 `tracing::warn`,UI 不打扰
5. **`MemoryConfig.working_memory_limit`** 默认 1000 条是否合理?——q-code 设计文档用 50,本设计用 1000。实施时跑一次模拟数据决定

### 显式不在本设计范围

- 多用户/团队/云端同步
- GUI / TUI
- Skill 热加载
- 多模态
- 向量检索
- 工作流 DSL

这些走方向 D 远景。
