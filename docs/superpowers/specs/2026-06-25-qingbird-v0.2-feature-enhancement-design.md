# qingbird-code V0.2 设计规格

> V0.1.0 restructuring 完成后的功能完善阶段

## 1. 概述

基于 V0.1.0（DeepSeek-only 重构）的稳定骨架，进入功能完善阶段。目标：让 agent 能干更多活 + 让交互体验更好 + 让编排更智能。

**三个优先级方向：**
1. 🥇 补齐工具集 — 目前仅 4 个工具（read_file / write_file / execute_command / search_code），agent 能力边界太窄
2. 🥈 CLI/TUI 体验打磨 — 进度指示 / 彩色输出 / Markdown 渲染 / TUI 滚动
3. 🥉 Subagent/编排增强 — 步骤上下文传递 / 角色差异化 / 部分失败恢复

## 2. 设计方案

### 2.1 新工具 (Iteration 1)

遵循现有 `capability/tools/` 目录结构，每个工具一个文件，实现 `Tool` trait，注册到 `ToolRegistry`。

#### list_dir (L0)

```rust
// struct ListDirTool — src/capability/tools/fs.rs (暂放, 或独立文件)
//
// 参数:
//   path: string, required — 目录路径
//   recursive: boolean, optional, default false — 是否递归
//
// 输出: 目录内容列表
//   [DIR]  subdir/
//   [FILE] main.rs    2.3 KB
//   [FILE] lib.rs     1.1 KB
//
// 实现: 用 tokio::fs::read_dir 拉目录项, WalkDir 处理递归
```

#### glob (L0)

```rust
// struct GlobTool — src/capability/tools/fs.rs
//
// 参数:
//   pattern: string, required — glob 模式, 如 "**/*.rs"
//   root: string, optional, default "." — 搜索根目录
//
// 输出: 匹配文件路径列表（每行一个）
//
// 实现: 用 globwalk crate 或手写简易匹配
// 注意: search_code 是内容搜索 (正则), glob 是文件名匹配 —— 正交的工具
```

#### http_get (L1)

```rust
// struct HttpGetTool — src/capability/tools/http.rs
//
// 参数:
//   url: string, required — 目标 URL
//   timeout_secs: u64, optional, default 30 — 超时秒数
//
// 输出: 状态码 + 响应内容 + Content-Type
//   HTTP 200 (text/html; charset=utf-8)
//   <html>...
//
// 风险 L1: 网络访问, 但仅 GET 且可控 URL
// 实现: reqwest::Client, 复用 Cargo.toml 已有 reqwest 依赖
// 注意: 复用 main.rs 中已有的 reqwest client (http_client.rs),
//       不额外新建 HTTP 连接池
```

#### edit_file (L1)

```rust
// struct EditFileTool — src/capability/tools/edit.rs
//
// 参数:
//   path: string, required — 文件路径
//   old_string: string, required — 旧文本 (需唯一匹配)
//   new_string: string, required — 新文本
//
// 输出: 替换位置 + 行数
//
// 实现:
//   1. 读文件
//   2. 搜索 old_string
//   3. 结果: 唯一匹配 → 替换; 未找到 → 报错提示精确内容;
//      多匹配 → 报错显示匹配数 + 前 3 处上下文, 建议 unique 化
//   4. 写回文件
//
// 设计决策: 不要备份; 如果 agent 改错可以再调 write_file 恢复
// (write_file 不依赖 edit_file, 互不影响)
```

### 2.3 Subagent/编排增强

#### 步骤间上下文传递 (Iteration 1)

```rust
// src/application/orchestrator.rs (或新文件 orchestrator/context.rs)
//
// 概念: 每步完成时, 从 Blackboard 提取关键信息注入下一步
// 实现: HashMap<String, String> — 由上一步的 Executor 写入
//   - "last_file_path": "src/main.rs"
//   - "last_result_summary": "Found 3 search results"
//   - "error_context": "File not found"
//
// 在 step 的 TaskSpec 中携带 context 字段
// 当前 step 能读取上一步的 context, 但不能跨多步
//
// 最小化改动: 只在 Blackboard 上加 context: HashMap, 
// Orchestrator 里插入 context 传递逻辑
```

#### 角色差异化 (Iteration 2)

```rust
// src/common/types.rs — 扩展现有 Role 枚举
//
// 角色:
// - Analyst: 搜索 + 读取 + 分析, 不写文件 / 不执行命令
//   capabilities: [ReadFile, SearchCode, WebFetch, LlmReasoning]
//
// - Executor: 命令执行 + 文件操作
//   capabilities: [ExecuteCommand, WriteFile, ReadFile]
//
// - Reviewer: 验证 + 反馈 (只读)
//   capabilities: [ReadFile, SearchCode, LlmReasoning]
//
// Orchestrator::execute() 里根据 step.tool 自动选角色:
//   "search_code" / "read_file" / "http_get" → Analyst
//   "execute_command" / "write_file" / "edit_file" → Executor
//   "llm_reasoning" → Generalist (保持默认)
```

#### 部分失败恢复 (Iteration 2)

```rust
// src/application/orchestrator.rs
// 
// 改造 execute() 的层执行循环:
// - 当前: 一步 Err → 整个任务 Err
// - 改为: 一步 Err → retry 1 次 (同层可重试)
//         retry 仍 Err → 记入 failed_steps, 继续下一层
// - 最终: 报告成功/失败步数
//
// 风险: 但有些步骤失败后续步骤无法继续 (依赖断了)
// 设计决策: 只对 L0/L1 步骤做自动恢复; L2+ 失败仍全局中止
```

## 5. ADR

### ADR-V0.2-001: 工具不备份

`edit_file` 不自动创建 `.bak` 备份。如果 agent 改错内容，可以调 `write_file` 从原始内容恢复。减少了文件 IO 和路径处理的复杂度。

### ADR-V0.2-002: 进度指示用 stderr

Spinner 输出到 stderr（非 stdout），保持 stdout 干净。遵循 spec B2 §3.5 stdout/stderr 协议。

### ADR-V0.2-003: 小文件先行

新工具放在独立文件中（非合并进已有大文件），便于维护和单元测试。文件命名按功能域：`fs.rs`（文件系统操作）、`http.rs`（网络请求）、`edit.rs`（文件编辑）。
