## 0. 一句话定义

在 spec A 跑通的 provider 抽象之上，加 3 个新能力：交互层 6 个斜杠命令通过 `SlashCommand` trait + `CommandRegistry` 注册；首次配置向导通过 `WizardStep` trait + `Wizard` 状态机；通用 SelectList widget 通过 `SelectItemSource` trait。**核心零硬编码命令名 / 步骤 / 选项**。

---

## 1. 背景 & 动机

### 1.1 spec A 留下的延伸需求

spec A 解决"provider 抽象"——但用户用 eflow 还差 3 件事：

1. **首次启动不知道怎么配** —— 用户 4 家 LLM 选哪家、KEY 在哪填、模型怎么选
2. **运行时想切模型 / 切 profile** —— 当前 Concierge 只能接任务输入，没有"运行时切状态"的命令
3. **未来要加新命令 / 新向导步骤** —— 必须 trait + 注册，**不能写死**

### 1.2 设计哲学（核心原则）

**轻量化内核 + 高标准要求**（用户 2026-06-17 明确）：

- 核心 crate 不写死任何命令名、步骤名、选项名
- 加新斜杠命令 = 写 1 个 `impl SlashCommand` + 1 行 `register()`
- 加新向导步骤 = 写 1 个 `impl WizardStep` + 1 行 `register()`
- 加新选项数据源 = 写 1 个 `impl SelectItemSource`

### 1.3 v1.2 现状（债）

```
src/interaction/
├── mod.rs              # pub use InteractionLayer
├── layer.rs            # InteractionLayer trait（v1.2 F2 抽的）
├── cli.rs              # CLI 实现（v1.2）
└── tui.rs              # TUI 实现（v1.2 F3-F5）—— 只接 Char/Backspace/Enter/Ctrl+C/Esc
                        # EnableMouseCapture 已 enable 但事件被吞
                        # 没有 SelectList widget
                        # 没有斜杠命令分发
                        # 没有向导
```

## 2. 目标

### 2.1 核心目标

1. **首次启动半自动**（用户已确认）：检测无 `~/.eflow/config.yaml` → 提示"是否进入向导" → 读 stdin 决定
2. **6 个斜杠命令可输入**（`/model /profile /lang /level /help /quit`）：通过 `SlashCommand` trait 注册，核心零硬编码
3. **向导 7 步可走完**（欢迎 → 语言 → 厂商 → 协议（仅自定义）→ KEY → 模型 → 确认）：通过 `WizardStep` trait 注册
4. **SelectList widget 多模交互**：输入序号 / ↑↓ 键 / 鼠标滚轮 / 鼠标点击 / Enter / Esc
5. **bare TUI 模式**：未配置时启动仍能看界面 + 收到 LLM 错误提示

**关键设计决策**：

1. **`interaction/slash/` 和 `interaction/wizard/` 独立目录**—— 6 个斜杠命令和 7 个向导步骤都是 6/7 个文件，加新命令 / 步骤 = 加文件 + 1 行 `register`，**主注册表只做 dispatch**。
2. **`interaction/widgets/` 抽出来**—— SelectList 是向导和运行时斜杠命令**共用的选择 UI**（`/model` 切模型也是 SelectList）。
3. **`Concierge` 不直接调具体命令**—— Concierge 接收 `String`（用户输入），**判断是否以 `/` 开头**，是 → 调 `CommandRegistry::dispatch()`；不是 → 当作普通任务派发。

### 3.2 trait 定义（核心 3 个）

#### `SlashCommand` trait

```rust
// src/interaction/slash/mod.rs
#[async_trait]
pub trait SlashCommand: Send + Sync {
    fn name(&self) -> &'static str;
    fn help(&self) -> &'static str;
    fn parse_args(&self, raw: &str) -> Result<SlashArgs, EflowError>;
    async fn execute(&self, args: SlashArgs, ctx: &mut CommandContext) -> Result<SlashOutput, EflowError>;
}

pub struct CommandContext<'a> {
    pub concierge: &'a mut Concierge,
    pub router: &'a mut LlmRouter,
    pub i18n: &'a mut Locale,
}

pub struct SlashArgs(pub HashMap<String, String>);

pub enum SlashOutput {
    Text(String),                // /help 输出的多行文本
    ReloadRouter,                // /model 改了 routing，需要重新构造 LlmRouter
    Shutdown,                    // /quit
    OpenSubView(Box<dyn View>),  // /model 切模型时弹 SelectList 子视图
    NoOp,                        // /lang 改了 i18n 但 TUI 不需要重画
}

pub struct CommandRegistry {
    commands: HashMap<&'static str, Arc<dyn SlashCommand>>,
}

impl CommandRegistry {
    pub fn new() -> Self;
    pub fn register(&mut self, cmd: Arc<dyn SlashCommand>);
    pub fn dispatch(&self, raw: &str) -> Option<(&'static str, SlashArgs)>;
    pub fn get(&self, name: &str) -> Option<&Arc<dyn SlashCommand>>;
    pub fn list(&self) -> Vec<(&'static str, &'static str)>;  // 给 /help 用
    pub fn required_register(&mut self, required: &[&'static str]) -> Result<()>;
}
```

**`required_register`**：main.rs 启动时调用，传入 6 个必须注册的命令名，缺失 → 启动失败。这条**确保未来有人忘记 register 时不沉默 panic**。

**`parse_args` 默认实现**：按空格 split 成 `HashMap<String, String>`（key 是位置 `arg0/arg1`，value 是 token）。命令可 override（如 `/level` 要校验枚举值）。

**`SlashOutput::OpenSubView`**：让命令能"打开子视图"——`/model` 切模型时**先弹 SelectList 让用户选 model id**，选完才调 `LlmRouter`。这个机制让所有"运行时选择"复用 SelectList。

#### `WizardStep` trait

```rust
// src/interaction/wizard/mod.rs
#[async_trait]
pub trait WizardStep: Send + Sync {
    fn id(&self) -> &'static str;                // "language" / "provider" / ...
    fn title(&self) -> &'static str;             // 显示在 wizard header
    fn render(&self, area: Rect, buf: &mut Buffer, state: &WizardState);
    fn on_key(&mut self, key: KeyEvent, state: &mut WizardState) -> StepAction;
    fn is_complete(&self, state: &WizardState) -> bool;
    fn next_step(&self) -> Option<Arc<dyn WizardStep>>;
    fn on_enter(&mut self, state: &mut WizardState);
    fn on_exit(&mut self, state: &mut WizardState);
}

pub enum StepAction {
    Stay,
    Next,
    Prev,
    Cancel,
}

pub struct Wizard {
    steps: Vec<Arc<dyn WizardStep>>,
    current: usize,
    state: WizardState,
}

pub struct WizardState {
    pub config: PartialConfig,
    pub providers: Vec<ProviderConfig>,
    pub skip_protocol_step: bool,
    pub fetch_failed: bool,
}
```

**关键设计决策**：
- `WizardState` 跨步共享，**每步只关心自己读写哪些字段**
- `next_step` 返回 `Option<Arc<dyn WizardStep>>`—— **状态机本身是数据**，**不是写死的 `match` 链**
- `skip_protocol_step` 让"preset vs 自定义"分叉——preset 跳过 protocol 步

#### `SelectItemSource` trait

```rust
// src/interaction/widgets/select_list.rs
#[async_trait]
pub trait SelectItemSource: Send + Sync {
    async fn items(&self) -> Result<Vec<SelectItem>, EflowError>;
}

pub struct SelectItem {
    pub label: String,
    pub value: String,
    pub hint: Option<String>,
    pub disabled: bool,
}

pub struct SelectList {
    source: Arc<dyn SelectItemSource>,
    selected: usize,
    scroll_offset: usize,
    viewport_height: usize,
}

impl SelectList {
    pub fn new(source: Arc<dyn SelectItemSource>) -> Self;
    pub fn render(&self, area: Rect, buf: &mut Buffer);
    pub fn on_key(&mut self, key: KeyEvent) -> SelectAction;
    pub fn on_mouse(&mut self, mouse: MouseEvent) -> SelectAction;
    pub fn selected_item(&self) -> Option<&SelectItem>;
}

pub enum SelectAction {
    Stay,
    Up,
    Down,
    PageUp,
    PageDown,
    Confirm(usize),
    Cancel,
}
```

**关键设计决策**：
- `items()` 是 async trait method——**model 列表异步拉取、preset 列表 sync 也能用**
- `on_key` / `on_mouse` 分离——TUI 主循环根据事件类型分发
- **焦点感知**：widget 不知道自己焦点位置，**TUI 主循环**根据鼠标点击坐标 + `widget.area()` 判断焦点在哪个 widget，**再 dispatch 鼠标事件**

### 3.3 6 个内置斜杠命令

| 命令 | 行为 | 输出 |
|---|---|---|
| `/model` | 弹 SelectList 子视图（数据源：当前 provider 的 model 列表 = `preset_models` + 拉取缓存）；选中后改 routing | `SlashOutput::OpenSubView` → 选完 → `ReloadRouter` |
| `/profile <name>` | 调 `Concierge::set_active_profile(name)`（v1.1 D1-D4 已实现） | `SlashOutput::Text(t!("status_profile_changed"))` |
| `/lang <zh-CN\|en-US>` | 调 `rust_i18n::set_locale` | `SlashOutput::Text(t!("status_lang_changed"))` |
| `/level <simple\|standard\|advanced>` | spec B1 **只挂空壳**：校验参数 + 输出 info message | `SlashOutput::Text(t!("info_level_pending_spec_c"))` |
| `/help` | 调 `registry_snapshot` 列出所有命令 + 帮助 | `SlashOutput::Text(...)` |
| `/quit` | 退出 eflow | `SlashOutput::Shutdown` |

### 3.4 7 个内置向导步骤

| step | 数据源 | 完成条件 |
|---|---|---|
| `welcome` | 静态文本 | 用户按 Enter |
| `language` | `SelectList { items: SUPPORTED_LOCALES }` | 选中语言 |
| `provider` | `SelectList { items: 4 个 preset 文件名 + "自定义" }` | 选中厂商 |
| `protocol` | `SelectList { items: OpenAI 兼容 / Anthropic 兼容 }` | 选中协议（仅自定义时显示） |
| `apikey` | prompt 输入框（**不是** SelectList） | KEY 非空 |
| `model` | `SelectList { items: 拉取的 model 列表 }` 或 prompt 输入框 | 选中 model 或手填 |
| `confirm` | 静态文本 + 摘要 | 用户按 Enter 确认 |

**关键设计决策**：
- `provider` step 的 SelectList 数据源是 `PresetFilesSource`，**从 `~/.eflow/providers/` + `docs/examples/providers/` 动态读**——首次启动时 `~/.eflow/providers/` 不存在，**fallback 到 `docs/examples/providers/`**（spec A 已经把 4 家示例文件放那里）
- `model` step 拉取失败时**自动切换到 prompt 输入态**——`SelectList::with_text_fallback()`
- `confirm` step 显示完整摘要（语言 / 厂商 / 协议 / KEY 掩码 / model），让用户最后核对

### 3.5 Concierge 改造（最小）

```rust
// src/application/concierge.rs 改
impl Concierge {
    pub async fn handle_input(&mut self, raw: String) -> Result<()> {
        if let Some(cmd) = raw.strip_prefix('/') {
            // 斜杠命令
            if let Some((name, args)) = self.command_registry.dispatch(cmd) {
                let cmd_impl = self.command_registry.get(name).unwrap();
                let output = cmd_impl.execute(args, &mut CommandContext { ... }).await?;
                self.apply_slash_output(output).await?;
            } else {
                self.messages.push(t!("err_unknown_slash_cmd", name = name).to_string());
            }
        } else {
            // 普通任务
            self.dispatch_task(raw).await?;
        }
        Ok(())
    }
}
```

**`apply_slash_output` 处理 `SlashOutput` 各种 variant**：
- `Text(s)` → 推到 messages 区
- `ReloadRouter` → 重新构造 `LlmRouter`（从最新的 ProviderConfig 列表）
- `Shutdown` → 发 `Event::SystemShutdown`
- `OpenSubView(view)` → TUI 主循环下一轮渲染时切换到子视图
- `NoOp` → 啥也不做

## 4. 组件

spec B1 的 7 个核心组件，每个独立可测。

### 4.1 C1. `SlashCommand` trait + `CommandRegistry`（`src/interaction/slash/mod.rs`）

**单一职责**：定义斜杠命令的 trait + 注册表 + dispatch。**不做**具体命令实现，不做 TUI 渲染。

**规则**：
- `register` 加重复 name → 第二个赢 + warn（v1.2 风格）
- `dispatch(raw)`: raw 是 `/model deepseek-v4-pro` 去掉 `/` 的 `model deepseek-v4-pro`
  - 按第一个空格 split → name + 剩余 args
  - 调对应命令的 `parse_args` 解析
  - name 不存在 → 返回 `None`（Concierge 显示"未知命令"）
- `parse_args` 默认实现：按空格 split 成 `HashMap<String, String>`（key 是位置，value 是 token）
- 6 个内置命令的 `parse_args` 可以 override（`/level` 接 `simple/standard/advanced` 3 个枚举值要校验）
- `required_register(&[&str])` 校验必需命令已注册，缺失 → 启动失败

**测试**：
- register + dispatch 成功
- register 同名 → 第二个赢
- dispatch 不存在的命令 → None
- parse_args 默认实现 + override 都跑通
- `list()` 返回所有注册命令（按 name 字母序）
- `required_register` 缺失必需命令 → Err

### 4.2 C2. 6 个内置命令（`src/interaction/slash/builtin/*.rs`）

每个命令是独立 `impl SlashCommand`。**关键设计决策**：

- `/level` 空壳不是"凑合"——**trait 早稳定，语义留给 C**，符合"不写死"原则
- `/model` 的 SelectList 数据源是 trait impl（不在 builtin.rs 里直接读 LlmRouter）——**核心零耦合**

**测试**（每命令 1-2 个）：
- `/model` 收到 `SlashOutput::OpenSubView` 且 view 是 SelectList
- `/profile` 调 Concierge::set_active_profile 且参数校验
- `/lang` 调 i18n 且参数在 SUPPORTED_LOCALES 列表内（不合法 → Err）
- `/level` 参数校验（3 个枚举值），空壳返回 info message
- `/help` 输出包含所有 6 个命令名
- `/quit` 返回 Shutdown

### 4.3 C3. `WizardStep` trait + `Wizard` 状态机（`src/interaction/wizard/mod.rs`）

**单一职责**：定义向导的 step 抽象 + 状态机。**不做**具体 step 实现。

**规则**：
- `Wizard::run(concierge, ctx)`: 从 `steps[0]` 开始循环，调用 `on_key` 直到 `is_complete` → 调 `next_step()` 推入下一步
- `on_enter` / `on_exit` 让 step 在进入/离开时初始化/收尾（如 `provider` step 离开时把 ProviderConfig 写 `state.providers`）
- `Esc` 键所有 step 统一处理 → `StepAction::Cancel`（不交给 step）
- `Backspace` 在 prompt 输入态交给 step（`apikey` step 需要 Backspace）

**测试**：
- 3 个 step 串行 → next_step 链正确
- preset provider 路径 → 跳过 protocol step
- Esc 取消 → Wizard::run 返回 Cancel
- `on_enter` / `on_exit` 在合适时机调用

### 4.4 C4. 7 个内置 step（`src/interaction/wizard/builtin/*.rs`）

**关键设计决策**：
- `provider` step 的 SelectList 数据源是 `PresetFilesSource`，**从 `~/.eflow/providers/` + `docs/examples/providers/` 动态读**——首次启动时 `~/.eflow/providers/` 不存在，**fallback 到 `docs/examples/providers/`**
- `model` step 拉取失败时**自动切换到 prompt 输入态**
- `confirm` step 显示完整摘要（语言 / 厂商 / 协议 / KEY 掩码 / model）

**测试**（每 step 1-2 个）：
- `welcome` 按 Enter → Next
- `language` 选 zh-CN → 写入 `state.config.core.language`
- `provider` 选 deepseek → 设置 `skip_protocol_step = true`
- `protocol` 仅在自定义时渲染
- `apikey` 输入 + Enter → 写入 `state.providers[0].api_key`
- `model` 拉取成功 → 渲染列表；拉取失败 → 渲染输入框
- `confirm` 按 Enter → 触发 `Wizard::finalize()`

### 4.5 C5. `SelectList` widget + `SelectItemSource` trait（`src/interaction/widgets/select_list.rs`）

**单一职责**：通用的多选一 widget + 数据源 trait。**不做**业务逻辑（不读 LlmRouter / 不读 Concierge）。

**规则**：
- `items()` async，**调用方负责超时**（widget 不阻塞）
- `on_key` 接 `Up/Down/PageUp/PageDown/Enter/Esc/Char(digit)`
  - `Char('1'-'9')` → 跳到第 N 项（**序号输入支持**）
- `on_mouse` 接 `MouseEventKind::ScrollUp/ScrollDown/MouseButton::Left`
  - `ScrollUp/ScrollDown` → 上下移动
  - `Left click on item N` → 跳到第 N 项
- `viewport_height` 控制可见项数，**滚动逻辑**当 selected 超出 viewport 时调整 `scroll_offset`
- `disabled` 项不可选（序号输入 / 鼠标点击都跳过）

**测试**（13 个）：
- 渲染：固定 5 项 + viewport=3 → 只显示 3 项
- 键盘 Up/Down 边界（首项 / 末项）
- 键盘 Char('3') → 跳到第 3 项
- 键盘 PageUp/PageDown
- 鼠标 ScrollUp/ScrollDown
- 鼠标点击第 N 项
- disabled 项被跳过
- 异步数据源：success / error / empty 3 种
- Esc → Cancel action
- Enter on disabled → Stay（不 Confirm）

### 4.6 C6. 首次启动检测（`src/main.rs` 改）

**单一职责**：检测 `~/.eflow/config.yaml` 是否存在，决定启动行为。

**规则**：
- 启动顺序：`LlmRouter::from_config` → 检测 config 文件存在性
  - **存在** → 跳过向导，直接进 TUI
  - **不存在** → 输出 `? 未找到 ~/.eflow/config.yaml，是否进入初始化向导？[Y/n]`（用 stderr 打印，不污染 stdout） → 读 stdin
    - `y` / `Y` / Enter（默认） → 启动 Wizard
    - `n` / `N` → 启动 bare TUI
- `eflow init` 命令 → **强制进 Wizard**（即使 config 存在也走向导，方便用户重新配置）

**关键设计决策**：
- 启动时是**阻塞读 stdin**——main.rs 启动后第一步就问，**不进 TUI**。
- 检测用 `path.exists()`，**不**读内容（避免解析错误时无法判断）。
- bare TUI 启动时**照常构造 LlmRouter**（如果 providers 目录空就退化到 env var），TUI 里 LLM 命令返回"未配置"错误。

**测试**：
- config 存在 → 跳过提示
- config 不存在 + stdin "y\n" → 启动 Wizard
- config 不存在 + stdin "n\n" → 启动 bare TUI
- `eflow init` → 强制 Wizard（无论 config 是否存在）
- `eflow` + EOF on stdin（CI 环境）→ 默认 Y（启动 Wizard）

### 4.7 C7. bare TUI 模式（`src/interaction/tui.rs` 改）

**单一职责**：在 TUI 主循环里检测"未配置"状态并显示简短警告 + 错误反馈。

**规则**：
- `TuiState` 加 1 个 `configured: bool` 字段（`LlmRouter` 构造时算出）
- header 区：`configured == false` 时显示 `⚠ 未配置 LLM provider`
- 用户在 prompt 输入普通任务（不是 `/` 开头） → 调 Concierge → Concierge 调 LlmRouter → 报 `err_no_llm_providers` → 推送到 messages 区
- **不**写硬编码错误文本——调 `t!("err_no_llm_providers")`（v1.2 已有的 i18n key）

**关键设计决策**：
- TUI **不**直接读 `~/.eflow/config.yaml` 存在性——**LlmRouter 内部已经知道**自己有没有 provider，**TUI 只查 trait 状态**：`router.is_empty()`（`LlmProviderRegistry` 加 1 个 `is_empty()` 方法，spec A 已有的类扩展 1 行）
- bare TUI 仍然能输入 `/help` / `/lang` / `/quit` 等**不依赖 provider** 的命令

**测试**：
- bare TUI 渲染时 header 包含 "⚠"
- LLM 命令返回 `err_no_llm_providers` 错误（mock Concierge）
- `/help` / `/quit` 在 bare TUI 仍可用

## 5. 数据流 & 错误处理

### 5.1 数据流

#### 启动时（main.rs）

```
1. main.rs 解析 CLI 参数
2. if arg == "init":
     config = load_existing_or_default()
     router = from_config(config, provider_dir)  // 走 spec A
     run_wizard(router) → 写 ~/.eflow/config.yaml + ~/.eflow/providers/*.yaml
     exit 0
3. config_path = ~/.eflow/config.yaml
   if !config_path.exists():
     print "未找到配置，是否进入初始化向导？[Y/n]" (stderr)
     read stdin
     if "n"/"N": goto bare_tui
   config = load(config_path) or default
4. router = from_config(config, provider_dir)  // spec A
5. concierge = Concierge::new(router, registry)
6. if bare_tui:
     tui = TuiBackend::with_bare_mode()  // header 显示警告
   else:
     tui = TuiBackend::new()
7. tui.run(concierge, events)
```

#### 向导运行时（`eflow init` 或首次启动确认 Y）

```
1. Wizard::run(steps: [welcome, language, provider, protocol?, apikey, model, confirm])
2. step 0: welcome → on_enter 显示欢迎 → 等 Enter → Next
3. step 1: language → on_enter 渲染 SelectList(SUPPORTED_LOCALES) → 用户选 → 写 state.config.core.language
4. step 2: provider → on_enter 渲染 SelectList(PresetFilesSource + "自定义")
   - 选中 preset: skip_protocol_step = true, state.providers.push(load_yaml(preset_file))
   - 选中 "自定义": skip_protocol_step = false
5. step 3: protocol? → 仅当 !skip_protocol_step 才显示 → 选 OpenAI/Anthropic 兼容 → 写 state
6. step 4: apikey → on_enter 渲染 prompt 输入框 → 用户输入 → 写 state.providers[0].api_key
7. step 5: model → on_enter 异步拉取 (timeout 5s)
   - 成功: 渲染 SelectList(models)
   - 失败: 渲染 prompt 输入框（自动 fallback）
8. step 6: confirm → on_enter 显示摘要 → 等 Enter → Wizard::finalize()
9. finalize():
   - 展开 ${ENV_VAR} (api_key)
   - 写 ~/.eflow/config.yaml
   - 复制 preset 文件到 ~/.eflow/providers/{id}.yaml (or 写自定义)
   - 输出 "配置完成，启动 eflow 开始使用"
   - exit 0
```

#### TUI 运行时（用户输入）

```
1. 用户在 prompt 输入字符串 s，按 Enter
2. Concierge::handle_input(s):
   if s.starts_with('/'):
     cmd_str = s.strip_prefix('/')
     dispatch(cmd_str):
       if found: cmd.execute(args, ctx) → SlashOutput
       else: 推 err_unknown_slash_cmd 到 messages
     apply_slash_output(output):
       Text(s) → push to messages
       ReloadRouter → 重新构造 LlmRouter
       Shutdown → 发 Event::SystemShutdown
       OpenSubView(view) → TUI 主循环下一轮切到子视图
       NoOp → continue
   else:
     dispatch_task(s) → Orchestrator
```

#### `/model` 命令特殊流程

```
1. /model 不带参数
2. ModelCmd::execute():
   - 获取当前 provider id (从 router.routing.strong)
   - 构造 ModelListSource { provider_id, router_ref }
   - 构造 SelectList::new(source)
   - 返回 SlashOutput::OpenSubView(Box::new(select_list))
3. TUI 主循环收到 OpenSubView → 渲染 SelectList
4. 用户选 / 按 Esc:
   - 选: TUI 把 selected value 通过 Concierge 改 router.routing.strong
   - Esc: 关闭子视图
5. TUI 切回主视图
```

### 5.2 错误处理表

| 边界 | 谁负责 | 失败时 |
|---|---|---|
| `~/.eflow/config.yaml` 不存在 | main.rs | 提示 + 读 stdin 决定 |
| `~/.eflow/providers/` 不存在 | PresetLoader (spec A) | 返回空 Vec（不报错） |
| 启动时 stdin EOF（CI 环境） | main.rs | 默认 Y（启动 Wizard） |
| 向导 step 1 `Esc` 取消 | Wizard | 退出向导 + 退出 eflow（不写部分配置） |
| 向导 step 2 `provider` 选自定义 | Wizard | 标记 `skip_protocol_step = false` |
| 向导 step 4 `apikey` 留空 | Wizard | 拒绝 Next（**不**允许空 KEY 继续） |
| 向导 step 5 `model` 拉取超时（5s） | Wizard | 自动切换到 prompt 输入态 + warn 国际化 |
| 向导 step 5 `model` 用户手填 | Wizard | 写入 `state.providers[0].default_model` |
| 向导 step 6 `confirm` `Esc` 取消 | Wizard | 退出向导 + 不写文件（**不**保存部分配置） |
| 向导 finalize 写文件失败（权限 / 磁盘满） | Wizard | 输出错误 + 退出码 1 + **不**启动 eflow |
| 斜杠命令 `parse_args` 参数非法 | 命令 | 返回 `Err`，Concierge 推 t!("err_arg_invalid") |
| 斜杠命令 execute 内部错误 | 命令 | 返回 `Err`，Concierge 推 t!("err_cmd_failed", msg) |
| `/model` 收到但当前 router 是 env var 退化路径（没自定义 provider） | ModelCmd | 输出 `t!("err_model_switch_no_provider")` |
| `/level` 收到（spec B1 空壳） | LevelCmd | 输出 `t!("info_level_pending_spec_c")` |
| bare TUI LLM 命令返回 `err_no_llm_providers` | Concierge | 推 t!("err_no_llm_providers") 到 messages |
| `SlashOutput::OpenSubView` 渲染失败 | TUI 主循环 | 不切换 + 输出 t!("err_subview_render_failed") |
| `CommandRegistry::required_register` 缺失必需命令 | main.rs | 启动失败 + 退出码 1 |

### 5.3 错误处理原则

1. **配置阶段宽容**：YAML 错、env var 缺、provider 不存在 → 跳过/warn
2. **运行阶段严格**：斜杠命令 / LLM 调用 → 返回明确错误
3. **向导不写半成品**：用户中途 Esc 取消 → 不写任何文件
4. **向导 finalize 原子写**：tmp 文件 + rename，避免半截配置污染 `~/.eflow/`

## 10. ADR

### ADR-0014：核心零硬编码斜杠命令

**状态**：已接受

**背景**：用户原话"轻量化内核，高标准要求"——6 个斜杠命令不能写死在 Concierge 的 `if input.starts_with('/help')` 里。v1.2 风格（`if-else` 链）扩展性差、加新命令改 core。

**决策**：spec B1 起
- `SlashCommand` trait + `CommandRegistry` 注册表
- 6 个内置命令各 1 个 `impl` 文件
- main.rs 启动时 `register()` + `required_register()` 校验
- 核心代码零硬编码命令名 / 帮助文本 / 参数 schema
- 加新命令 = 写 1 个 `impl SlashCommand` + 1 行 `register()`

**代价**：每个命令多写 1 个文件（30-50 行）；与 v1.2 风格不同。

**收益**：
- v2.0 GUI 团队可以独立加命令（共享 `SlashCommand` trait impl）
- spec B2 CLI 契约复用同一份 trait impl
- 加新命令零改 core
- 类型安全（trait 编译期校验）

### ADR-0015：核心零硬编码向导步骤

**状态**：已接受

**背景**：7 个 wizard 步骤本质不同（语言选择 vs 厂商选择 vs KEY 输入 vs 模型选择），但**所有"步骤 = 状态机的一格"**。写死 `match step_id` 不可扩展。

**决策**：spec B1 起
- `WizardStep` trait + `Wizard` 状态机
- 7 个内置 step 各 1 个 `impl` 文件
- 加新步骤 = 写 1 个 `impl WizardStep` + 1 行 register

**代价**：每 step 多写 1 个文件。

**收益**：未来加"检查环境变量" / "导入旧配置"等步骤零改 core。

## 12. 已知偏差（v1.3 spec B2 头脑风暴阶段发现）

**发现时间**：2026-06-17 spec B2 段 3 用户提"渲染核心引擎"原则后
**原则**：eflow 渲染原则 —— TUI 显示层零硬编码；所有线条 / 颜色 / 文本内容来自 render 引擎的指令

**本 spec 违反位置**：

| 位置 | 违反 |
|---|---|
| `WizardStep::render(&self, area, buf, state)` trait 方法 | wizard step 直接调 ratatui API（`Paragraph::new` / `Style::default().fg(Color::...)`）——硬编码"画什么 + 怎么画" |
| `SelectList::render(&self, area, buf)` trait 方法 | 同样问题——SelectList 直接画线/颜色 |
| 7 个 wizard step impl（`wizard/builtin/*.rs`） | 每步 `render` 方法里硬编码文案 + 颜色 |
| `TuiBackend::render(state, f)` 函数 | v1.2 已有 `Color::Cyan` / `Color::Yellow` 等硬编码主题 |

**修复 spec**：v1.3.3 spec D — 渲染引擎 + TUI 重构

**修复方案**（spec D 范围内详化）：

1. `WizardStep` trait 的 `render()` 改 `view_model()` —— wizard 只描述"我是哪步 + 显示什么数据"
2. `SelectList` 同上 —— 改为 `view_model()`
3. 新增 `RenderEngine` trait + `DrawCommand` enum —— core 发指令，ratatui 退到"只执行 DrawCommand"的角色
4. TUI / GUI / 未来 web = 不同 `RenderBackend` impl，共用同一份 ViewModel + DrawCommand
5. 7 个 wizard step impl 重写 —— 不再调 ratatui，直接构 ViewModel
6. `TuiBackend` 重写 —— 把 `Vec<DrawCommand>` 翻译成 ratatui 的 `Paragraph::new` / `Style::default().fg(...)`

**对 spec B1 实施的影响**：

- spec B1 实施时 **WizardStep / SelectList / TuiBackend 三个 trait/函数的 render 部分用"临时实现"满足当前设计** —— 直接调 ratatui API 让 spec B1 能跑通
- spec B1 任务量从 12 → **14-15 tasks**（多了 2-3 个 view_model 重构占位 + 偏差标注）
- spec B1 文档的"组件"和"任务清单"会在 spec D 实施时**回填修订**（diff 模式标注"已被 spec D 重构"）
- **不影响 spec B1 的 50 个单元测试逻辑** —— 测试断言"行为"（用户输入 → step 转移 → state 更新），不直接断言 ratatui 像素

**对 spec D 范围的影响**：

- spec D 接手时**不再讨论"是否要重构"** —— spec B1 已经确认"必须重构"
- spec D 范围：8-10 tasks（RenderEngine trait + DrawCommand enum + 2 个 RenderBackend impl + 7 个 wizard step 改 view_model + TuiBackend 重写 + 1 个 theme.rs + 文档）

**Why 这条偏差**：
用户在 spec B2 段 3 明确："TUI 界面显示层禁止出现硬编码糊弄。显示界面就只起到显示出画面的功能，所有的动作动画功能传达都是需要从渲染引擎发出指令，显示界面只是按照指令在什么地方显示出什么而已，不许出现设计时这里需要一条线，然后硬编码给写一条线。那样毫无意义。同理 渲染引擎所渲染出的也同样时软件核心所给出的功能要求，也是同样禁止出现硬编码糊弄"。

**How to apply**：
- v1.3 实施分 4 个小版本：**v1.3.0 (A) / v1.3.1 (B1) / v1.3.2 (B2 + C 并行) / v1.3.3 (D)**
- B1 实施时走"临时硬编码"路径，D 实施时全量重构
- B1 的 50 个单元测试**保留作为 B1 行为契约**，D 实施时**新增** RenderEngine 单元测试（不替换 B1 测试）
- ADR-0017 写进 spec D 文档"核心零硬编码渲染"
