## 0. 一句话定义

v1.4 spec D 重构 eflow 的渲染管线：所有 UI 显示从「wizard step / SelectList / TuiBackend 直接调 ratatui API」改为「核心发意图 → RenderEngine 翻译 → RenderBackend 机械执行」的电脑主机三层架构。

**ADR-0017 核心**：核心零硬编码渲染——所有显示必须走完整路径，禁止任何「先 hardcode 跑通 + TODO」的临时路径。

### 1.2 atlas 失败教训（来自 EFO analysis 文档）

`atlas` 是 eflow 之前的独立失败项目，TUI 重构 3 次后停掉。3 个根因：

1. **架构层**：`tui.go` 是 God Object（2137 行，35 个字段）
2. **流程层**：TDD 漏了状态机/竞态/快照/集成/e2e 测试
3. **概念层**：渲染管线缺少 Scene 抽象

EFO analysis 文档原话：

> "Atlas 踩的坑不是 TDD 的问题，不是 Go 的问题，不是 TUI 框架的问题——是架构分层没有严格锁死的问题。"

**eflow 当前风险**：v1.3.1 阶段「先 hardcode + 留 TODO」就是 atlas 的种子。如果 v1.4 不根除，未来会重演「重构 3 次」失败。

### 1.3 电脑主机架构（用户核心思路）

类比：

- 软件核心 = 主机（wizard/业务逻辑）
- RenderEngine = 显卡（翻译「画什么」为「具体怎么画」，硬编码颜色/前缀/边框）
- RenderBackend = 显卡驱动（机械执行 ratatui）
- 屏幕 = 终端显示器

**3 个不变量**：

- 任何东西要画到屏幕，必须走「核心 → RenderEngine → RenderBackend」完整路径
- RenderBackend 不知道「这是 wizard 还是 SelectList」，只知道「画 DrawCommand」
- 核心不知道 ratatui（只输出 ViewModel）

## 2. 核心架构

### 2.1 三层数据流

```
[软件核心]    -- 表达意图 -->    [RenderEngine]    -- 翻译 -->    [RenderBackend]    -- 执行 -->    屏幕
ViewModel                          DrawCommand                       ratatui
纯数据                             纯指令                            机械
```

### 2.2 4 条铁律

1. **核心层零 ratatui import**：`grep -r "ratatui" src/interaction/wizard/ src/interaction/widgets/` 必须 0 结果
2. **RenderBackend 零业务知识**：TuiBackend 不知道是 wizard 还是 SelectList，只知道画 DrawCommand
3. **Modal 走完整路径**：`/model` 弹 SelectList 也走 `FrameViewModel::Modal`
4. **TuiBackend::run() 不准业务判断**：TUI 只接事件 + 调 RenderEngine，不准 `if key == '/model'`

### 2.3 改造涉及文件

| 文件 | 改造方式 |
|------|----------|
| `tui.rs` | `render()` 改 `execute(Vec<DrawCommand>)`；`run()` 移 `/model` 硬处理 |
| `select_list.rs` | `render()` 改 `view_model() -> SelectListVM` |
| `wizard/builtin/*.rs` × 7 | `render()` 改 `view_model() -> StepViewModel` |
| `wizard/mod.rs` | `run()` 改完整 event loop（支持 ↑↓/Tab 切焦点） |
| 3 处 v1.3.1 TODO 注释 | 全部删除 |

## 3. 组件抽象

### 3.1 ViewModel（核心层产物）

**3 种 ViewModel + 3 个子 VM**：

#### `FrameViewModel`（整屏视图）

枚举类型：

- `FullScreen(ScreenVM)` — 全屏显示一种内容
- `Modal { background: ScreenVM, popup: ScreenVM }` — 主屏 + 弹窗叠合

#### `ScreenVM`（屏幕内「一段」内容）

枚举类型：

- `Wizard(StepViewModel)` — 向导步骤
- `SelectList(SelectListVM)` — 列表 widget
- `Main(MainViewModel)` — TUI 主屏 4 段布局

#### `StepViewModel`（wizard 一步）

字段：

- `title: String`
- `lines: Vec<LineVM>` — 显示的多行文本
- `input: Option<InputFieldVM>` — 输入框（如有）
- `hints: Vec<KeyHint>` — 底部按键提示
- `focused_field: usize` — 当前焦点 field 索引

#### `SelectListVM`（列表 widget）

字段：

- `title: String`
- `items: Vec<ListItemVM>` — 列表项
- `selected: usize` — 当前选中项
- `scroll_offset: usize` — 滚动偏移

#### `MainViewModel`（TUI 主屏 4 段）

字段：

- `header: HeaderVM` — 头部 4 段布局的 header
- `messages: Vec<MessageVM>` — 中部消息列表
- `status: String` — 状态栏
- `prompt: String` — 输入框

每个 `*VM` 是纯数据 struct（无方法，或只有 `to_string()` 之类纯函数）。**ViewModel 不包含坐标 / area 信息**——布局由 RenderBackend 在 `execute(commands, area, buf)` 时根据当前 frame 的 area 决定，ViewModel 只表达「要显示什么」不表达「在哪里显示」。

### 3.2 DrawCommand（RenderEngine 产物）

5 类指令：

```
DrawCommand::Text      { content: String, style: TextStyle }
DrawCommand::Block     { border: BorderToken, title: String }
DrawCommand::Span      { prefix: String, content: String, style: TextStyle }
DrawCommand::Line      { commands: Vec<Span> }
DrawCommand::ClearArea
```

`TextStyle` 字段由 RenderEngine 硬编码：直接用 `ratatui::style::Color`（仅 `src/interaction/render/` 目录可 import ratatui）。

### 3.3 RenderEngine trait

```rust
pub trait RenderEngine: Send + Sync {
    fn render(&self, vm: &FrameViewModel) -> Vec<DrawCommand>;
}
```

**1 个默认实现**：`DefaultRenderEngine`

- 主色 cyan（`Color::Cyan`）
- 警告 red（`Color::Red`）
- 缓存 yellow（`Color::Yellow`）
- 选中前缀 "▶ "
- 未选中前缀 "  "
- 边框 Rounded

未来 v1.5+ 可加 `HighContrastEngine`、`DarkEngine`，**v1.4 不交付**。

### 3.4 RenderBackend trait

```rust
pub trait RenderBackend {
    fn execute(&mut self, commands: &[DrawCommand], area: Rect, buf: &mut Buffer);
}
```

**1 个实现**：`TuiBackend`（ratatui）。它读 DrawCommand 调 `Paragraph::new` / `List::new`，**但只机械执行，不决定颜色**。

## 6. ADR-0017 提案

### 6.1 标题

**ADR-0017: 核心零硬编码渲染**

### 6.2 状态

草案

### 6.3 背景

v1.3.1 阶段实施 spec B1 时，3 处文件直接调 ratatui API（违反「零硬编码」原则），留 3 处 TODO 注释待 v1.4 重构。`atlas` 项目（独立失败项目）TUI 重构 3 次失败教训：架构分层没锁死是根因。

### 6.4 决策

eflow 渲染管线采用电脑主机三层架构：

1. **软件核心**（wizard/SelectList/TuiBackend 4 段布局）输出 **ViewModel**（纯数据）
2. **RenderEngine**（默认 impl：DefaultRenderEngine）翻译 ViewModel → **DrawCommand**（硬编码颜色/前缀/边框）
3. **RenderBackend**（默认 impl：TuiBackend = ratatui）机械执行 DrawCommand 画到屏幕

**铁律**：

- 核心层零 ratatui import
- RenderBackend 零业务知识
- Modal 走完整路径
- 禁止任何「先 hardcode + TODO」临时路径

### 6.5 后果

**好的**：

- 换 RenderBackend（egui/web）只重写最后一层
- 测试可在不启动 ratatui 的情况下验证 ViewModel/RenderEngine
- 业务代码（wizard step）零渲染知识

**不好的**：

- 多一层转换（ViewModel → DrawCommand）
- 调试时需要跟踪多层数据流

### 6.6 备选方案（已否决）

- **A. 2 层架构**（ViewModel → RenderBackend）—— 失去多 backend 灵活性
- **B. 4 层架构**（+ ThemeToken 抽象）—— 用户已否决（「不要搞得看不懂」）
- **C. 保留 v1.3.1 临时 hardcode** —— 重蹈 atlas 失败

## 10. 不在 v1.4 范围（明确排除）

为避免范围蔓延，以下显式排除：

1. **GUI RenderBackend impl**（egui/iced）—— v1.5+ 单独 spec

7. **跨 RenderBackend 的 ViewModel 一致性测试**—— v1.4 只有一个 RenderBackend，v1.5+ 多 backend 时再加
