# qingbird-code V0.2.0 设计规格 — Agent 核心循环 + 多 Provider

> 基于 V0.1.0 (DeepSeek-only) → V0.2.0 (ReAct 循环 + DeepSeek/Ollama/OpenAI/Anthropic)

## 1. 概述

### 1.1 目标

将 qingbird-code 从"规划先行"的 D-E-F 管线升级为"观察-行动-观察"的 ReAct 循环，同时引入多 Provider 支持（DeepSeek 深度优化、Ollama 本地实测、OpenAI/Anthropic 占位），并拆分为 Cargo workspace 多 crate 架构。

### 1.2 动机

- **学习目的**：深入理解 ReAct 循环 vs D-E-F 管线的设计差异
- **DeepSeek 专属**：利用 DeepSeek V4 系列的 thinking mode、reasoning_content、硬盘缓存等特性做深度优化
- **本地可测**：Ollama 提供零成本的本地真链路测试能力
- **架构扎实**：拆 workspace 学习 crate 边界管理

### 1.3 参考来源

- qingbirdd (v0.1.8) 架构 & ReAct 循环实现
- [DeepSeek API 官方文档](https://api-docs.deepseek.com/) — 2026/06 最新版
- qingbird-code V0.1.0 现有代码

## 2. Crate 拆分

### 2.1 从单 crate → 5 crates workspace

```
crates/
├── qbird-code           # 二进制入口 (main.rs + cli + tui + interaction)
├── qbird-code-models    # 共享类型，零外部依赖
├── qbird-code-infra     # 基础设施 (Provider/LLM/缓存/配置/记忆/事件)
├── qbird-code-tools     # 工具系统 (Tool trait + ToolRegistry + 内置工具)
└── qbird-code-agents    # 🆕 Agent 核心 (ReAct 循环 + Subagent + Prompt)
```

### 2.2 依赖方向（严格自下而上，禁止循环）

```
qbird-code-models          ← 零依赖
    ↑
qbird-code-infra           ← 依赖 models
    ↑
qbird-code-tools           ← 依赖 models + infra
    ↑
qbird-code-agents          ← 依赖 models + infra + tools
    ↑
qbird-code (binary)        ← 依赖以上全部
```

### 2.3 各 crate 职责

| Crate | 一句话描述 | 现有代码来源 |
|---|---|---|
| `qbird-code-models` | 所有共享数据类型 | `src/common/types.rs` + `error.rs` |
| `qbird-code-infra` | LLM/Provider 调用、缓存、配置、记忆、事件 | `src/infrastructure/` 全部 |
| `qbird-code-tools` | 工具定义、注册、执行 | `src/capability/tools/` |
| `qbird-code-agents` | ReAct 循环核心、Subagent 管理、Prompt 组装 | **新建** |
| `qbird-code` | CLI 入口、TUI、交互层 | `src/main.rs` + `src/interaction/` + `src/cli/` |

### 2.4 拆分原则

1. 每个 crate 只做一件事，一句话能说清
2. 上层不能反向依赖下层（编译即文档）
3. 先有第二个实现再提 trait — infra 里的 Provider 现在有 4 个实现，可以提 trait
4. 文件拆分遵循：单文件超过 300 行考虑拆，超过 500 行必须拆

## 3. 多 Provider 设计

### 3.1 Provider 架构

```
crates/qbird-code-infra/src/
├── providers/
│   ├── mod.rs           # Provider trait + ProviderKind 枚举
│   ├── deepseek.rs      # 🔥 完整实现（双协议 + thinking + 缓存优化）
│   ├── ollama.rs        # 完整实现（OpenAI 兼容协议）
│   ├── openai.rs        # 基础实现（占位，无 key 不测）
│   └── anthropic.rs     # 基础实现（占位，无 key 不测）
├── http_client.rs       # 统一 HTTP 客户端 (reqwest)
├── router.rs            # Provider 路由（按名称 + 模型选择）
├── config.rs            # 配置加载
└── ...
```

### 3.2 ProviderKind 枚举

```rust
pub enum ProviderKind {
    DeepSeek,
    OpenAI,
    Anthropic,
    Ollama,
}
```

### 3.3 协议类型

```rust
pub enum ProtocolKind {
    /// OpenAI 兼容协议 — DeepSeek (主), OpenAI, Ollama 共用
    /// endpoint: /chat/completions
    OpenAICompatible,
    /// Anthropic 原生协议 — Anthropic 专用
    /// endpoint: /messages
    Anthropic,
}
```

### 3.4 Provider trait

```rust
#[async_trait]
pub trait Provider: Send + Sync {
    fn kind(&self) -> ProviderKind;
    fn protocol(&self) -> ProtocolKind;
    fn model(&self) -> &str;
    fn base_url(&self) -> &str;

    /// 构建符合该 Provider 协议的请求体
    fn build_request(
        &self,
        messages: &[Message],
        tools: &[ToolDefinition],
        config: &RequestConfig,
    ) -> serde_json::Value;

    /// 解析响应体为统一格式
    fn parse_response(&self, body: &serde_json::Value) -> Result<ChatResponse>;

    /// 解析流式 chunk
    fn parse_stream_chunk(&self, chunk: &[u8]) -> Result<Option<ChatChunk>>;
}
```

    base_url_anthropic: "https://api.deepseek.com/anthropic"  # Anthropic 协议

## 4. DeepSeek 专项设计

> 以下全部基于 [DeepSeek 官方文档](https://api-docs.deepseek.com/) 2026/06 最新版。

### 4.1 模型版本

| 模型名 | 状态 | 说明 |
|---|---|---|
| `deepseek-v4-pro` | ✅ 当前主力 | 最强模型，thinking 默认开启 |
| `deepseek-v4-flash` | ✅ 当前 | 更快更便宜 |
| `deepseek-chat` | ⚠️ 2026/07/24 弃用 | 兼容别名 → v4-flash 非思考模式 |
| `deepseek-reasoner` | ⚠️ 2026/07/24 弃用 | 兼容别名 → v4-flash 思考模式 |

### 4.2 Thinking Mode

#### 控制参数

| 控制项 | OpenAI 格式 | Anthropic 格式 |
|---|---|---|
| 开关（默认开启） | `{"thinking": {"type": "enabled/disabled"}}`，通过 `extra_body` 传入 | `{"thinking": {"type": "enabled/disabled"}}` |
| 强度控制 | `reasoning_effort: "high"/"max"` | `output_config: {"effort": "high"/"max"}` |
| effort 映射 | low/medium → high, xhigh → max | 同左 |
| Agent 请求 | 自动设为 max | 同左 |

#### 不支持参数

thinking 模式下以下参数**不报错但不生效**：
- `temperature`
- `top_p`
- `presence_penalty`
- `frequency_penalty`
- `logprobs`
- `top_logprobs`

### 4.3 reasoning_content 处理（关键！）

DeepSeek 返回的消息中 `reasoning_content` 与 `content` 同级。

#### 多轮对话规则

| 场景 | 规则 |
|---|---|
| assistant **无** tool_calls | `reasoning_content` 无需传入下一轮，API 会忽略 |
| assistant **有** tool_calls | `reasoning_content` **必须完整回传**后续所有请求，否则返回 400 错误 |

#### 推荐做法

直接 `messages.push(response.choices[0].message)`，DeepSeek 返回的 message 对象包含 `content`、`reasoning_content`、`tool_calls` 三个字段。简单且不出错。

#### Message 类型

```rust
pub struct Message {
    pub role: MessageRole,       // system / user / assistant / tool
    pub content: String,
    /// DeepSeek thinking — 仅 assistant 角色时有值
    pub reasoning_content: Option<String>,
    /// 工具调用 — 仅 assistant 角色时有值
    pub tool_calls: Option<Vec<ToolCall>>,
    /// 工具调用结果 ID — 仅 tool 角色时有值
    pub tool_call_id: Option<String>,
}
```

### 4.4 硬盘缓存（KV Cache）

#### 机制

DeepSeek 对所有用户**默认开启**，无需应用层代码修改。缓存是"尽力而为"的，不保证 100% 命中。

#### 缓存落盘时机

1. **请求结束位置落盘**：用户输入结束位置 & 模型输出结束位置各生成一个缓存前缀单元
2. **公共前缀检测落盘**：多请求有公共前缀时自动提取
3. **按固定 token 间隔落盘**：长输入/输出按间隔截取

#### 命中规则

| 第一次 | 第二次 | 命中？ |
|---|---|---|
| A + B | A + B + C | ✅ 命中 A+B |
| A + B | A + C | ❌ 不命中。但系统提取公共前缀 A 落盘 |
| → A 已落盘 | A + D | ✅ 命中 A |

#### 缓存追踪

从 `usage` 中提取：

```rust
pub struct CacheUsage {
    pub prompt_cache_hit_tokens: u64,   // 缓存命中 tokens
    pub prompt_cache_miss_tokens: u64,  // 缓存未命中 tokens
}
```

#### 应用层优化策略

虽然缓存是自动的，但我们可以通过消息排序最大化命中率：

```
消息结构（固定前缀在前）:
  [system_prompt] [tool_definitions] [user_msg_1] [assistant_msg_1] ... [user_msg_n]
   └── 固定前缀 ────┘
   所有请求共享 → 高缓存命中率
```

- ✅ system prompt 放最前面且内容固定
- ✅ 工具定义紧随 system prompt
- ✅ 不在前缀中插入时间戳等动态内容
- ❌ 不要在 system prompt 前面加任何动态消息

### 4.5 DeepSeek 双协议

| 协议 | base_url | endpoint | 适用场景 |
|---|---|---|---|
| OpenAI 兼容 | `https://api.deepseek.com` | `/chat/completions` | 主力使用 |
| Anthropic 兼容 | `https://api.deepseek.com/anthropic` | `/messages` | Claude Code 等 Anthropic 生态工具接入 |

**Anthropic 格式关键差异：**
- `cache_control` 被忽略（DeepSeek 用自动硬盘缓存）
- `image` / `document` 类型不支持
- `thinking` → `budget_tokens` 被忽略，仅 `type` 有效
- `output_config` → 仅 `effort` 支持
- Anthropic API 传入不支持的模型名 → 自动映射到 `deepseek-v4-flash`
- Claude 模型名映射：`claude-opus*` → v4-pro，`claude-haiku*` / `claude-sonnet*` → v4-flash

## 5. ReAct 循环设计

### 5.1 核心概念

ReAct (Reason-Act) 循环替代 D-E-F 管线。核心变化：**LLM 自己决定每一步做什么**，不再由 Orchestrator 预先分解任务。

### 5.2 要删除的代码

| 文件 | 原因 |
|---|---|
| `src/capability/decisioner.rs` | 不再需要"一次性规划" |
| `src/capability/feedbacker.rs` | 反馈融入循环的 nudge 机制 |
| `src/capability/blackboard.rs` | 替换为消息管线 |
| `src/application/orchestrator.rs` | 不再需要"任务分解+依赖分层" |
| `src/application/concierge.rs` | 重构为轻量入口 |

### 5.3 循环流程

```
输入: 用户消息
  │
  ▼
┌─────────────────────────────────────────────┐
│  loop (最多 max_iterations 次)               │
│                                              │
│  1. 安全自检                                  │
│     ├─ 迭代上限 → wind-down 总结文本后退出     │
│     ├─ 用户中断 → Interrupted                │
│     └─ 自动上下文压缩 (token 超阈值时)         │
│                                              │
│  2. LLM 调用                                  │
│     ├─ 组装消息 (system + history + user)     │
│     ├─ 附加工具定义                            │
│     ├─ DeepSeek: 处理 thinking 参数           │
│     └─ 发起 HTTP 请求                         │
│                                              │
│  3. 响应处理                                  │
│     ├─ 解析 content / reasoning_content       │
│     ├─ 死循环检测 (见 5.5)                     │
│     └─ 决定 TurnResult:                       │
│         ├─ Complete → 返回结果                 │
│         ├─ ToolCalls → 执行工具 → 继续循环      │
│         └─ 无工具调用 → 追加内容 → 继续         │
│                                              │
│  4. 工具执行 (见 5.4)                          │
│     ├─ 只读工具 → 批量并行                      │
│     └─ 写入工具 → 逐个串行                      │
│                                              │
│  5. Nudge 检测                                │
│     ├─ 连续 5 轮只读 → 提醒"该写代码了"         │
│     └─ 说完成但 Todo 未完成 → 提醒补充          │
│                                              │
│  6. 结果追加到消息历史 → 回到步骤 1             │
└─────────────────────────────────────────────┘
  │
  ▼
输出: AgentResult
```

### 5.4 工具执行策略（三级）

```rust
enum ExecutionStrategy {
    /// 子 agent 全部并行 — spawn_subagent 调用
    ParallelSubagents,
    /// 只读工具批量 — Read, Glob, Grep, WebFetch, WebSearch 等一次并发执行
    BatchedReadonly,
    /// 写入工具串行 — Write, Edit, Bash 等逐个执行
    Sequential,
}
```

| 策略 | 适用工具 | 并发模型 |
|---|---|---|
| `ParallelSubagents` | `spawn_subagent` 产生的 agent | `FuturesUnordered` |
| `BatchedReadonly` | Read, Glob, Grep, WebFetch, WebSearch, TaskList | 同一 batch 内并发 |
| `Sequential` | Write, Edit, Bash, 未知工具 | 逐一执行 |

### 5.5 死循环检测 (Doom Loop)

```rust
pub struct DoomLoopDetector {
    recent: VecDeque<String>,    // 最近 20 条指纹
    nudge_count: usize,          // 已检测次数
}

// 指纹: "tool_name:args_hash"
// 检测周期: 1/2/3 步
// 阈值: 重复 3 次 → 判定死循环
```

**三级升级**：

| 次数 | 动作 | 行为 |
|---|---|---|
| 1st | Redirect | 注入提醒到消息: "换个思路" |
| 2nd | Notify | 更强提醒: "建议推翻重来" |
| 3rd | ForceStop | 清理指纹历史，返回失败结果 |

### 5.6 Nudge 机制

| 触发条件 | Nudge 内容 |
|---|---|
| 连续 5 轮仅只读工具 | "你已经连续读取了很多文件，考虑开始写代码" |
| 并发起了 3+ 子 agent 但没等结果 | "你有 N 个后台任务在运行，建议先检查结果" |
| completion_nudge (LLM 说完成但无工具调用) | "如果有待办事项未完成，请先完成" |
| 迭代接近上限 | "你只有最后几轮了，请尽快给出结论" |

### 5.7 LoopState

```rust
/// 循环中跨迭代保持的可变状态
pub struct LoopState {
    pub iteration: usize,
    pub consecutive_no_tool_calls: usize,    // 连续无工具调用轮次
    pub consecutive_reads: usize,            // 连续只读轮次
    pub doom_detector: DoomLoopDetector,
    pub completion_nudge_sent: bool,
}
```

### 5.8 关键类型

```rust
pub enum TurnResult {
    Continue,                                    // 继续循环
    ToolCalls { tool_calls: Vec<ToolCall> },    // 需要执行工具
    Complete { content: String, status: Option<String> },  // 任务完成
    MaxIterations,                               // 达到迭代上限
    Interrupted,                                  // 用户中断
}

pub struct AgentResult {
    pub content: String,
    pub messages: Vec<Message>,   // 完整消息历史（用于会话持久化）
    pub status: AgentStatus,
    pub usage: UsageStats,
}

pub struct UsageStats {
    pub prompt_tokens: u64,
    pub completion_tokens: u64,
    /// DeepSeek 专属：缓存命中 tokens
    pub cache_hit_tokens: u64,
    /// DeepSeek 专属：缓存未命中 tokens
    pub cache_miss_tokens: u64,
}
```

## 6. Subagent 管理

### 6.1 从 D-E-F Subagent → 独立 ReAct Agent

现有 Subagent 每个只执行 D-E-F 管线中的一个步骤。新版：每个 subagent 是一个**完整的 ReAct 循环实例**。

### 6.2 Subagent 角色

| 角色类型 | 工具白名单 | 说明 |
|---|---|---|
| `CodeExplorer` | Read, Glob, Grep, LSP | 只读探索 |
| `CodeWriter` | Read, Write, Edit, Glob, Grep | 代码修改 |
| `Planner` | 无工具（纯推理）| 任务规划 |
| `Generalist` | 全部工具 | 通用 |

### 6.3 Subagent 运行模式

```rust
pub struct SubagentConfig {
    pub name: String,
    pub role: SubagentRole,
    pub allowed_tools: Vec<String>,     // glob 匹配，如 "read_*"
    pub model: Option<String>,          // 独立模型绑定 (None=继承主 agent)
    pub max_iterations: usize,          // 默认 25
    pub system_prompt: String,
}
```

Subagent 作为异步 Tokio 任务运行（非独立进程），通过 `SubagentEventBridge` 向主 agent 的 UI 推送进度事件。

## 7. 数据流

### 7.1 完整请求生命周期

```
User Input (CLI/TUI)
  │
  ▼
Binary (qbird-code)
  │── 加载配置 (qbird-code-infra)
  │── 获取会话历史 (如果有)
  │── 路由到 Agent
  │
  ▼
MainAgent.run(message)
  │
  ├── 1. Prompt 组装
  │      system_prompt + tool_defs + message_history + user_message
  │
  ├── 2. ReAct Loop (qbird-code-agents)
  │      │
  │      ├── 安全自检 (上限/中断/压缩)
  │      ├── LLM 调用 → Provider (qbird-code-infra)
  │      │     ├── DeepSeek: thinking + reasoning_content 处理
  │      │     ├── Ollama: OpenAI 兼容协议
  │      │     └── 返回 ChatResponse
  │      ├── 死循环检测
  │      ├── 工具执行 (qbird-code-tools)
  │      │     ├── 只读批量并行
  │      │     └── 写入逐一串行
  │      ├── Nudge 检测
  │      └── 结果追加 → 继续循环
  │
  └── 3. 返回 AgentResult
       ├── content + messages + usage
       └── 事件推送
```

## 8. 错误处理

### 8.1 Provider 级

| 错误 | 处理 |
|---|---|
| 网络超时 | 指数退避重试 (max 3 次) |
| 401/403 鉴权失败 | 立即返回错误，不重试 |
| 400 (reasoning_content 缺失) | 自动补充后重试 1 次 |
| 429 频率限制 | 等待 Retry-After 后重试 |
| 5xx 服务端错误 | 指数退避重试 |

### 8.2 Agent 级

| 错误 | 处理 |
|---|---|
| 死循环 (3 次检测) | 强制停止，返回失败 |
| 上下文超限 | 触发自动压缩 |
| 迭代上限 | wind-down 总结后返回 |
| 用户中断 | 立即返回 Interrupted |
