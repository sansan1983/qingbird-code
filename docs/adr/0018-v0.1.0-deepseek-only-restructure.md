# ADR-0018: V0.1.0 DeepSeek 专属重构

- **状态**: 已采纳
- **日期**: 2026-06-24
- **版本**: v0.1.0

## 背景

eflow v1.0~v1.4 积累了大量多 Provider 抽象代码：`ProtocolKind` 枚举（OpenAI/Anthropic 两种兼容协议）和 `tier` 路由（strong/medium/light → 不同 provider），以及 4 家预设 provider 的 YAML 文件。实际使用中只用到 DeepSeek（甚至只用 `deepseek-chat` 一个模型）。

多 Provider 抽象增加了大量代码和测试维护成本，LLM 调用链路在多层包装后 debug 困难，同时 `eflow` 命名与另一工具重名造成混淆。

## 决策

### 1. 项目重命名

`eflow` → `qingbird`（青鸟），二进制名 `qingbird`，库名 `qingbird_code`。

### 2. DeepSeek 专属化

删除所有多 Provider 抽象：

- `ProtocolKind` 枚举 → 简单字符串协议标识
- `ProviderConfig` struct → `serde_json::json!()` 动态构造
- `ModelEntry`/`pick_model()`/`list_models_endpoint` → 删除
- `supports_prefix_cache()`/`retry_params()` 等 trait 默认方法 → 删除
- LlmProvider trait 9 方法 → 3 方法（chat / chat_stream / name）
- 4 家预设 provider → 零预置
- `~/.qingbird/providers/*.yaml` → 删除
- 配置文件 `llm.providers` + `llm.routing` → `llm.deepseek`
- 配置查找路径 `eflow.yaml` → `qingbird.yaml` / `~/.qingbird/config.yaml`
- `check_llm_configured()` 检查 provider 文件 → 检查 `DEEPSEEK_API_KEY` 环境变量或配置文件
- 旧 `eflow init` provider 向导 → 需要重写但先保持旧向导骨架

### 3. 被删除的主要类型和模块

| 删除项 | 用途 | 替代 |
|--------|------|------|
| `ProtocolKind` | 区分 OpenAI/Anthropic 协议 | `&'static str` |
| `ProviderConfig` | provider YAML 元数据 | `serde_json::json!()` |
| `ModelEntry` | 模型列表条目 | 删除 |
| `ModelTier` | strong/medium/light 路由 | 删除 |
| `pick_model()` | 按 tier 选模型 | 删除 |
| `supports_prefix_cache()` | 缓存兼容性判断 | 删除 |
| `rate_limit_counters` | 单 provider 限流 | 删除 |
| `~/.qingbird/providers/*.yaml` | provider 元数据配置 | 删除 |
| `llm.providers` / `llm.routing` | 配置文件字段 | `llm.deepseek` |
| 4 家预设 list_models | provider 模型列表 | 删除 |

### 4. 保留的架构

- 四层架构（交互/编排/能力/基础设施）不变
- Concierge → Orchestrator 编排层不变
- Decisioner → Executor → Feedbacker 管线段不变
- 3 层记忆（Working/Project/User）不变
- Event Bus 机制不变
- TUI (ratatui) 交互层不变
- Subagent 池不变

### 5. 不在 V0.1.0 范围

- 配置向导 `qingbird init` 的深重构（旧向导骨架保留但不可用）
- `~/profiles/` 目录清理（保留旧 profile 系统骨架）
- Test 全面重写（部分测试用 dummy key 验证编译通过，但 LLM 路径预期失败）
- README.en.md / CHANGELOG.md / CONTRIBUTING.md / AGENTS.md 归档（未改动）
- 旧 `doc/` 目录的 spec/plan 文档（未改动）

## 后果

### 正面

- 代码减少约 300 行 LLM 抽象层
- 编译更快（删除了 4 家 provider 的 heavy 依赖）
- Debug 路径变短（chat → deepseek，不再经过 routing/middleware）
- 配置简化：一个环境变量即可运行

### 负面

- 失去多 Provider 灵活性（如需其他 provider 需改源码）
- package name 变更影响已有 CI/安装脚本（如果存在）

## 未来方向

- V0.1.1: `qingbird init` 配置向导重写
- V0.1.2: 清理 /docs/eflow 遗留文档，归档新文档结构
